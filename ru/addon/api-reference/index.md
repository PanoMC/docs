# Справочник API фронтенда

Эта страница перечисляет **каждое имя хука, слот представления, событие жизненного цикла, помощник навигации и экспорт `@panomc/sdk`**, которые может использовать UI вашего аддона, — в одном месте. Это **справочная страница**, а не руководство.

**Впервые здесь? Четыре вещи, каталогом которых является эта страница, простыми словами:**

- **hook (хук)** = именованное место в странице хоста, где отрисовывается ваш компонент.
- **view slot (слот представления)** = как хук, но элементы упорядочены (по числу приоритета) и могут скрываться или показываться по отдельности.
- **lifecycle event (событие жизненного цикла)** = момент во время загрузки страницы, в который вы можете выполнить код.
- **navigation helper (помощник навигации)** = API для добавления или редактирования ссылок в меню сайта или боковой панели администратора.

::: tip Как читать эту страницу
Это набор плотных справочных таблиц, сгруппированных по области API (§1–§10). Вы не читаете её сверху вниз — вы прыгаете к разделу для той штуки, которую подключаете. Каждый раздел открывается одним простым предложением, говорящим, для чего он и **где работает** (тема, панель или обе). Если термин в ячейке таблицы незнаком, он почти наверняка определён во врезке «Понятия, которые предполагает эта страница» чуть ниже — прочитайте эту врезку один раз сначала.
:::

::: tip Понятия, которые предполагает эта страница (прочитайте один раз, ~60 секунд)
Эта страница переиспользует горстку слов десятки раз каждое. Вот каждое одним простым предложением:

- **Хост** — работающий фронтенд Pano, который загружает и запускает JavaScript вашего аддона. Это одно из двух приложений: **тема** или **панель** (следующий пункт). «Хост» здесь никогда не означает сервер, который вы арендуете.
- **Тема против панели** — два фронтенда. **Тема** — публичный сайт, который видят игроки, по адресу `/`. **Панель** — административная панель, по адресу `/panel`. Ваш аддон может работать в обоих, и каждый предоставляет **разные** API. `pano.isPanel` сообщает вашему коду, в котором из них он находится.
- **Хранилище Svelte** — значение, на которое можно подписаться. В `.svelte`-компоненте поставьте `$` перед ним, чтобы читать его реактивно (`$myStore`). В обычном JS вызовите `store.subscribe(fn)` и держите функцию, которую он возвращает, чтобы позже прекратить слушать (запушьте эту функцию в `this._unsubscribers`, см. §1). Несколько API ниже «возвращают хранилище». ([Документация Svelte: хранилища](https://svelte.dev/docs/svelte/stores))
- **`load(event)`** — необязательная функция, которую может экспортировать страница или хук. Хост вызывает её, пока готовит страницу. `event` описывает запрос (его URL, параметры, куки). См. врезку «три вида `load()`» ниже.
- **props** — значения, которые Svelte-компонент получает от того, что его отрисовало. Какой бы объект ни вернул ваш `load()`, он становится props вашего компонента.
- **SSR / гидрация** — SSR («серверный рендеринг») означает, что при **первом** открытии посетителем страницы её HTML строится на сервере; последующие навигации строятся в браузере. **Гидрация** — это когда Svelte привязывает этот построенный на сервере HTML в браузере, чтобы кнопки и т. д. стали интерактивными.
- **голый спецификатор / внешний** — *голый спецификатор* — это путь импорта без `./` или `/`, например `import { x } from 'svelte'`. Ваша сборка оставляет разрешённые из них **внешними** — она **не** копирует их в JS вашего аддона; хост поставляет их во время выполнения. См. §10.
- **узел права доступа** — строка права доступа вроде `x.y.z`, которая решает, кому разрешено что-то видеть или делать. Список узлов и того, как аддоны определяют свои собственные, см. в [Справочнике Backend API](/ru/addon/backend-reference/).
:::

::: tip Три вида `load()` на этой странице
Слово `load` появляется в трёх разных ролях. Они **не** взаимозаменяемы — путать их — классический баг первой недели:

| Где | Вы пишете | Оно получает | Что означает его возврат |
|---|---|---|---|
| **Страница** (§2) | `export function load(event)` в модуле страницы | `event` (запрос) | props компонента страницы (четыре особых ключа вроде `pageTitle` извлекаются для хоста) |
| **Хук** (§3) | `export function load(event)` в модуле хука | `event` (запрос) | props компонента хука; верните `{ hookOptions: { invisible: true } }`, чтобы скрыть хук |
| **Обработчик жизненного цикла** (§4, §6, §7) | `(data, event) => { … }`, переданный в вызов `onLoad(...)` / `lifecycle.on(...)` | `data` (данные страницы, иногда изменяемые) **и** `event` | обычно ничего — вы читаете, или для помеченных событий подкручиваете `data` на месте |
:::

Если вы хотите увидеть эти API, использованные в настоящем аддоне, начните с [Разработки фронтенда](/ru/addon/frontend/), которая строит UI Shoutbox шаг за шагом. Всё здесь — это поверхность, из которой черпает та страница.

Для половины вашего аддона на Kotlin у этой страницы есть родственник: **[Справочник Backend API](/ru/addon/backend-reference/)** делает ту же работу для поверхности бэкенда — жизненного цикла плагина, базы данных, эндпоинтов, прав доступа и событий.

::: tip Откуда они берутся
Вы пишете `import { X } from 'svelte'`, но ваша сборка никогда не встраивает Svelte в вывод вашего аддона — она оставляет этот импорт **внешним**, и работающий сайт Pano (хост) поставляет одну общую копию во время выполнения. Вот почему разрешены только определённые импорты (см. §10). Дерево `pano.*`, документированное ниже, внедряется в ваш плагин как `this.pano`; модули `@panomc/sdk` — это замороженный список импортов в конце этой страницы. (Полный механизм: [Архитектура](/ru/addon/architecture/).)
:::

## 0. Объект `pano`

Всё достигается через объект `pano`, который хост внедряет в ваш плагин как `this.pano`. Один флаг живёт наверху; остальное свисает с `pano.ui.*`.

| Свойство | Тип | Что это |
|---|---|---|
| `pano.isPanel` | boolean | `true`, когда ваш код выполняется внутри административной **панели**, `false` внутри **темы**. Используйте его, чтобы выполнять разные регистрации в каждой — то есть `if (this.pano.isPanel) { /* panel registrations */ } else { /* theme registrations */ }`. |

Панель и тема предоставляют **разные** деревья `pano.ui` — помощник, существующий в одном, может не существовать в другом. Поэтому на протяжении этой страницы каждый раздел говорит, где живут его члены: **тема + панель**, **только тема** или **только панель**. Когда весь раздел односторонний (как §4, только тема), об этом говорят его заголовок и врезка наверху.

## 1. Контракт точки входа плагина

Ваш `src/main.js` экспортирует по умолчанию класс, расширяющий `PanoPlugin` (из `@panomc/sdk`). Хост создаёт один его экземпляр, внедряет `this.pano` и вызывает методы жизненного цикла ниже.

Наименьший возможный файл-точка входа — это просто вот это:

<!-- src/main.js — the whole file; your registrations go inside onLoad() -->
```js
import { PanoPlugin } from '@panomc/sdk';

export default class extends PanoPlugin {
  onLoad() {
    // your register(...) calls from sections 2–5 go here
  }
}
```

Каждую таблицу дальше читать легче, если вы представляете этот скелет.

| Член | Вид | Назначение |
|---|---|---|
| `onLoad()` | метод (переопределение) | Вызывается один раз после загрузки плагина. Выполняйте все свои **регистрации** здесь (вызовы `register(...)` из разделов 2–5 ниже). `this.pano` доступен. **Не трогайте `this.pano` в конструкторе** — он внедряется только прямо перед выполнением `onLoad()`. |
| `onUnload()` | метод (переопределение) | Вызывается, когда плагин демонтируется. Отмените всё, что не должно задержаться (например, `pano.ui.page.unregister(...)`). |
| `this.pano` | свойство | Внедрённый объект API, документированный на этой странице. |
| `this.context` | свойство | Обычный объект, где ваш плагин держит состояние, которым хочет поделиться со своими компонентами (например, настройки, которые вы получили). |
| `this.setContext(partial)` | метод | Копирует ключи `partial` в `this.context` (как `Object.assign` — **поверхностное** слияние, на один уровень вглубь) и уведомляет всё, что подписано на контекст, что он изменился. |
| `this._unsubscribers` | массив | Пушьте сюда функции отписки от хранилищ (см. пункт про хранилища во врезке Понятий); хост выполняет их все, когда плагин уничтожается, чтобы ваши подписки не текли. |

Две функции приходят из `@panomc/sdk` вместе с базовым классом:

| Экспорт | Назначение |
|---|---|
| `viewComponent(importer)` | **Правило: всегда пишите `viewComponent(() => import('./X.svelte'))`, когда передаёте компонент любому вызову `register`** — никогда голый `() => import(...)` и никогда сам компонент. (Он прикрепляет правильные `mount`/`hydrate`/`unmount` общей среды выполнения для хоста; из-за этого «водопровода» обёртка обязательна.) |
| `getPanoContext()` | Возвращает текущий контекст хоста Pano. Вы можете это игнорировать, если только вам не нужен сырой контекст хоста из кода, который не является методом `PanoPlugin` (у обычного метода вместо этого есть `this.context`) — редко нужно. |

::: warning `onContextUpdate` не вызывается
Если класс, который вы создали из boilerplate, содержит метод `onContextUpdate()`, **удалите его — ни один хост его не вызывает.** Это мёртвый код; не стройте поведение на нём. Используйте `onLoad()` для настройки и подписки на хранилища (см. врезку Понятий) для реакции на изменения.
:::

## 2. Страницы — `pano.ui.page` (тема + панель)

Используйте их, чтобы регистрировать собственные целые страницы — под темой (`/…`) или панелью (`/panel/…`).

| Вызов | Назначение |
|---|---|
| `pano.ui.page.register(options)` | Зарегистрировать страницу по пути. |
| `pano.ui.page.unregister(path)` | Удалить страницу, которую вы зарегистрировали (используется для очистки — см. [Разработку фронтенда](/ru/addon/frontend/)). |
| `pano.ui.page.isPluginPage(path)` | `true`, если какой-то плагин зарегистрировал этот путь. |

**`register(options)`:**

| Опция | Тип | Значение |
|---|---|---|
| `path` | string | Маршрут (см. формы путей ниже). |
| `component` | `viewComponent(...)` | Компонент страницы. |
| `systemLayout` | string | Обернуть вашу страницу в один из встроенных макетов хоста (имена перечислены ниже). Используйте `MainLayout` для обычной страницы; остальные имена соответствуют конкретным разделам хоста (например, `ProfileLayout`, `SettingsLayout`). |
| `layout` | `viewComponent(...)` | Использовать собственный компонент макета вместо встроенного. |
| `resetLayout` | boolean | Отрисовать **без** шапки, боковой панели или подвала хоста — вашему компоненту достаётся вся страница. («Chrome» — общее слово для этого окружающего UI хоста.) |
| `permission` | string | Узел права доступа (строка права доступа вроде `x.y.z` — список см. в [Справочнике Backend API](/ru/addon/backend-reference/)), требуемый для просмотра. Если у текущего пользователя его нет, страница отрисовывается как **404**. |

**Формы путей:**

| Форма | Пример | Совпадает с |
|---|---|---|
| литеральная | `/shoutbox` | ровно этот путь |
| динамический сегмент | `/shout/[id]` или `/shout/:id` | один сегмент, захваченный как параметр — читайте его в вашем `load(event)` как `event.params.id` |
| catch-all | `/docs/[...rest]` | оставшиеся сегменты (должен быть финальным сегментом) |
| regex | `re:/shout/\d+` | шаблон должен совпасть со **всем** путём, а не только частью — хост оборачивает его в `^…$` («полностью заякорен») за вас |

Модуль страницы может также **экспортировать `load(event)`** (см. врезку «три вида `load()`» ближе к началу). `event` описывает запрос — его URL, параметры и куки. Какой бы объект вы ни вернули, он передаётся вашему компоненту страницы как его **props**. Четыре особых ключа **удаляются** из этого возвращённого объекта и используются chrome хоста вместо передачи как props:

- `pageTitle` — заголовок, показываемый для страницы (например, вкладка браузера).
- `breadcrumbs` — след «хлебных крошек», который хост отрисовывает над страницей.
- `sidebar` — боковая панель для этой страницы.
- `sidebarProps` — props, передаваемые этой боковой панели.

**Имена `systemLayout` — тема:** `AppLayout`, `AuthLayout`, `MainLayout`, `ProfileLayout`, `ThemeSettingsLayout`, `TicketsLayout`.

**Имена `systemLayout` — панель:** `AddonDetailLayout`, `AddonsLayout`, `AppLayout`, `MainLayout`, `MigrationLayout`, `PermissionsLayout`, `PlayerDetailLayout`, `PlayersLayout`, `PostsLayout`, `ServerLayout`, `ServerSettingsLayout`, `SettingsLayout`, `TicketsLayout`, `TranslationsLayout`, `ViewLayout`.

::: tip Контрольная точка — зарегистрировалась ли моя страница?
После вызова `pano.ui.page.register({ path: '/your-path', component })` пересоберите свой аддон и перезагрузите сайт. Посещение `/your-path` теперь должно показать ваш компонент. Ничего нет? Проверьте, что `component` обёрнут в `viewComponent(() => import('./X.svelte'))` и что `register` выполнился внутри `onLoad()`.
:::

## 3. Хуки — `pano.ui.hook` (тема + панель)

**Хук** — это именованная дырка в странице хоста: всё, что вы регистрируете под именем этого хука, отрисовывается в том фиксированном месте. Хук — это плоский список компонентов под одним именем. (Слоты представлений в §4 добавляют упорядочивание и скрытие поверх этой же идеи.)

| Вызов | Где | Назначение |
|---|---|---|
| `pano.ui.hook.register(options)` | тема + панель | Смонтировать компонент в именованный хук. |
| `pano.ui.hook.get(name)` | тема + панель | Возвращает **хранилище** (см. врезку Понятий) компонентов, зарегистрированных для `name`. |
| `pano.ui.hook.setVisible(name, component, visible)` | только тема | Переключить видимость записи хука. Передайте **ту же ссылку на компонент**, что вы дали в `register`. |

**`register(options)`:**

| Опция | Тип | Значение |
|---|---|---|
| `name` | string | Имя хука (таблицы ниже). |
| `component` | `viewComponent(...)` | Компонент для монтирования. |
| `permission` | string | Отрисовывать только для пользователей, держащих этот узел права доступа. |
| `skipLoad` | boolean | Не выполнять `load()` компонента во время загрузки страницы. Используйте это, когда ваш хук получает собственные данные при монтировании, чтобы прогон во время загрузки страницы был напрасным или провалился. |
| `invisible` | boolean | Зарегистрировать, но начать скрытым. |

**Контракт `load()` / `hookProps` хука:** модуль компонента хука может экспортировать `load(event)`. Хост выполняет его во время загрузки страницы — на сервере при первом открытии страницы (SSR) и в браузере при навигации между страницами — и передаёт результат вашему компоненту как его props (этот объединённый объект props и есть `hookProps` компонента). Компонент может скрыть себя, вернув `{ hookOptions: { invisible: true } }` **из своего `load()`**.

::: tip Чтение имени хука
Имя хука грубо говорит, где оно отрисовывается. **Префикс** — это область: `theme:` = публичный сайт, `page:` = область контента страницы, `panel:` = административная панель. **Середина** называет страницу или таблицу. **Суффикс** называет место: `:top`, `:bottom`, `:content`, `:sidebar` или позицию в таблице вроде `:header:...` / `:row:...`. Не уверены точно, куда что-то приземляется? Зарегистрируйте компонент, отрисовывающий видимый маркер, пересоберите и посмотрите.
:::

## 4. Слоты представлений — `pano.ui.view` (только тема)

**Слот представления** — это именованный контейнер, который отрисовывает **упорядоченный по приоритету список** компонентов плагинов (дополнительные методы входа, дополнительные строки профиля и так далее). Как хук, но каждый элемент слота несёт `id` и `priority`, поэтому элементы можно по отдельности скрывать, переупорядочивать или заменять.

**Хуки против слотов представлений с одного взгляда:**

| | Хук (§3) | Слот представления (§4) |
|---|---|---|
| Упорядочивание | нет (плоский список) | по `priority` (выше отрисовывается первым) |
| id на элемент | нет | да (`id`) — позволяет скрыть/переместить/заменить один элемент |
| Где работает | тема + панель | **только тема** |

::: warning `pano.ui.view` / `pano.ui.sidebar` существуют только в теме
- **Строите для панели?** Пропустите весь этот раздел — панель вообще не предоставляет `view.register/hide/show/move/get/onLoad/load` или `pano.ui.sidebar`.
- **Исключение панели 1:** дополнительные строки в модальном окне редактирования игрока имеют собственный выделенный API — см. «Панель: строки модального окна редактирования игрока» ниже.
- **Исключение панели 2:** единственный член `pano.ui.view` панели — это `pano.ui.view.themes.editMenu` — см. §8.
:::

| Вызов | Назначение |
|---|---|
| `pano.ui.view.register({ viewId, id, component, priority })` | Добавить компонент в слот `viewId`. `priority` по умолчанию `10`; повторная регистрация того же `id` заменяет его. |
| `pano.ui.view.hide(viewId, id)` | Скрыть элемент, не удаляя его. |
| `pano.ui.view.show(viewId, id)` | Снова показать его. |
| `pano.ui.view.move(viewId, id, priority)` | Изменить приоритет элемента. |
| `pano.ui.view.get(viewId)` | Возвращает **хранилище** (см. врезку Понятий) видимых, упорядоченных элементов. |
| `pano.ui.view.onLoad(viewId, handler)` | Выполнять ваш обработчик каждый раз, когда загружаются данные этого слота. (Под капотом это событие жизненного цикла `theme:view:<viewId>:load` — см. §6.) |
| `pano.ui.view.load(viewId, event)` | Выполнить конвейер загрузки слота и получить разрешённые элементы (для страницы плагина, которая сама содержит слот). |

`pano.ui.sidebar.*` — это **алиас** того же реестра с теми же методами, за исключением того, что ключ контейнера — `sidebarId` вместо `viewId` (а `onLoad` запускает `theme:sidebar:<id>:load`).

**Форма элемента слота:** `{ id, component, priority, props? }`. Более высокий `priority` отрисовывается первым. **Нет** поля `permission` на элемент — элементы слота не фильтруются по правам доступа. Чтобы ограничить один, проверьте право доступа **внутри вашего компонента** — `import { hasPermission } from '@panomc/sdk/utils/auth'` — и ничего не отрисовывайте, если проверка не прошла. (Хуки и ссылки навигации *поддерживают* опцию `permission`.)

### Панель: строки модального окна редактирования игрока

У панели **нет** реестра слотов `pano.ui.view`. Её единственная точка расширения этого вида — дополнительные строки в модальном окне редактирования игрока — вместо этого имеет собственный выделенный API:

| Вызов | Назначение |
|---|---|
| `pano.ui.player.editModal.cardRows.edit(callback)` | Отредактировать список строк карточки, показываемых в модальном окне редактирования игрока. `callback` получает текущий массив строк; измените его на месте и верните. |
| `pano.ui.player.editModal.cardRows.get()` | Прочитать текущие строки карточки. |

Объекты строк отражают существующие строки модального окна; их точные поля здесь не перечислены — вызовите `cardRows.get()` (или `console.log` массив, который получает ваш колбэк `edit`), чтобы их осмотреть. Аддоны в стиле аватаров и social-login используют это, чтобы добавить строку в это модальное окно.

### Соглашения о приоритетах

Совпадайте с этими числами, чтобы ваши элементы приземлялись в разумном порядке **относительно других установленных аддонов**. Напоминание: **более высокий приоритет отрисовывается первым**, поэтому отрицательный `-100` намеренно ставит внедрения поддержки **последними**.

| Вид слота | Соглашение |
|---|---|
| строки модального окна редактирования игрока | `100` |
| строки карточки настроек | `105` |
| строки карточки профиля | `90` |
| альтернативные методы авторизации | `50` |
| внедрение поддержки | `-100` |
| всё остальное | `10` (по умолчанию) |

## 5. Навигация — `pano.ui.nav`

Добавляйте или редактируйте ссылки в меню сайта (тема) или боковой панели администратора (панель). Тема и панель предоставляют **разные** помощники.

**Тема (только тема):**

| Вызов | Назначение |
|---|---|
| `pano.ui.nav.site.editNavLinks(callback)` | **Синхронный.** Получает текущий массив ссылок; либо измените его на месте, либо верните новый массив. Результат заново сортируется хостом по `priority` каждой ссылки. |
| `pano.ui.nav.site.getNavLinks()` | Хранилище текущих ссылок навигации сайта. |
| `pano.ui.nav.profileDropdown.edit(callback)` / `.get()` | Отредактировать / прочитать элементы выпадающего меню профиля (это редактирует слот `navbar-profile-dropdown` из §4). |
| `pano.ui.nav.rightComponents.edit(callback)` / `.get()` | Отредактировать / прочитать компоненты справа в навбаре (это редактирует слот `navbar-right` из §4). |
| `pano.ui.nav.onLoad(handler)` | Подписаться на `theme:navbar:load` (событие жизненного цикла — см. §6). |

**Форма ссылки навигации темы** — `{ href, text, icon?, target?, startsWith, loginRequired?, permission?, priority? }`, поле за полем:

| Поле | Обязательно | Значение |
|---|---|---|
| `href` | да | Куда указывает ссылка, например `/shoutbox`. |
| `text` | да | Видимый ярлык. Если строка содержит `.`, она трактуется как ключ перевода и прогоняется через `_` (см. §9 / Локализацию); иначе показывается как есть. |
| `startsWith` | да | **Булево** значение, управляющее подсветкой активной ссылки. Когда `true`, ссылка подсвечивается всякий раз, когда путь текущего URL *начинается с* её `href` (так что `/shoutbox` остаётся подсвеченным на `/shoutbox/123`); когда `false`, подсвечивает только точное совпадение пути. |
| `icon` | нет | **CSS-строка класса** иконки, отрисовываемая на `<i>`, например `'fa-solid fa-comments'`. |
| `target` | нет | Стандартный `target` якоря, например `'_blank'` для новой вкладки. |
| `loginRequired` | нет | Если `true`, показывать ссылку только вошедшим пользователям. |
| `permission` | нет | Узел права доступа; показывать ссылку только пользователям, которые его держат. |
| `priority` | нет | Порядок сортировки среди ссылок (меньшее число идёт первым; ссылки без него сортируются последними). |

**Панель (только панель):**

| Вызов | Назначение |
|---|---|
| `pano.ui.nav.site.editNavLinks(async handler)` | **Асинхронный, обязан вернуть массив.** Редактирует ссылки главной боковой панели панели. |
| `pano.ui.nav.server.editNavLinks(async handler)` | Асинхронный, обязан вернуть массив. Редактирует ссылки боковой панели раздела сервера. |

::: warning Колбэки навигации панели обязаны вернуть массив
`editNavLinks` темы принимает изменение на месте, но `editNavLinks` панели (и `server.editNavLinks`) **асинхронны** и устанавливают список в то, что вы **вернёте** — забудьте `return`, и вы сотрёте меню. Это выглядит так:

```js
pano.ui.nav.site.editNavLinks(async (links) => {
  links.push({ href: '/panel/shoutbox', text: 'Shoutbox' });
  return links; // forget this line and the sidebar goes blank
});
```
:::

## 6. События жизненного цикла

События времени загрузки, которые хост запускает, пока готовятся данные страницы. Каждый обработчик имеет сигнатуру `async (data, event)` — `event` — это тот же объект-событие запроса, который получает ваш `load()` (§2), а `data` — данные страницы в процессе подготовки. В большинстве обработчиков вы просто **читаете** `data`; для помеченных событий ниже вы можете его **изменять** (см. столбец «Заметки по data»). Регистрируйте через сокращённые помощники ниже или через общий примитив:

| Вызов | Назначение |
|---|---|
| `pano.ui.lifecycle.on(name, handler)` | Подписаться на любое событие жизненного цикла по имени (тема + панель). |
| `pano.ui.lifecycle.execute(name, data, event)` | **Только тема.** Запустить жизненный цикл самому — например, ваш плагин отрисовывает собственную страницу входа и хочет, чтобы на ней выполнился жизненный цикл входа хоста (и обработчики других аддонов). `pano.ui.lifecycle` панели предоставляет только `on`. |

### События жизненного цикла темы

| Событие | Сокращение | Заметки по data |
|---|---|---|
| `theme:app:load` | `pano.ui.app.onLoad(h)` | — |
| `theme:navbar:load` | `pano.ui.nav.onLoad(h)` | — |
| `theme:profile:load` | `pano.ui.profile.onLoad(h)` | — |
| `theme:settings:load` | `pano.ui.settings.onLoad(h)` | — |
| `theme:tickets:load` | `pano.ui.tickets.onLoad(h)` | — |
| `theme:login:load` | `pano.ui.auth.login.onLoad(h)` | `data = { error, event }` — вы можете установить `data.error = '…'`; после выполнения вашего обработчика хост читает его и показывает на странице входа |
| `theme:register:load` | `pano.ui.auth.register.onLoad(h)` | `data = { error, username, event }` — вы можете установить `data.error` и `data.username`; хост читает их обратно |
| `theme:reset-password:load` | `pano.ui.auth.resetPassword.onLoad(h)` | — |
| `theme:activate:load` | `pano.ui.auth.activate.onLoad(h)` | `data = { token }` — `token` — это код активации из ссылки в письме пользователя (взят из URL) |
| `theme:activate-new-email:load` | `pano.ui.auth.activateNewEmail.onLoad(h)` | `data = { token }` — та же идея, для подтверждения нового email |
| `theme:renew-password:load` | `pano.ui.auth.renewPassword.onLoad(h)` | `data = { token }` — та же идея, для ссылки сброса пароля |
| `theme:post-detail:load` | `pano.ui.post.onLoad(h)` | — |
| `theme:support:load` | `pano.ui.support.onLoad(h)` | — |
| `theme:view:<viewId>:load` | `pano.ui.view.onLoad(viewId, h)` | запускается на каждый слот |
| `theme:sidebar:<id>:load` | `pano.ui.sidebar.onLoad(id, h)` | запускается на каждую боковую панель |

### События жизненного цикла панели

| Событие | Сокращение | Заметки по data |
|---|---|---|
| `panel:posts:load` | `pano.ui.posts.onLoad(h)` | — |
| `panel:addon-detail:load` | `pano.ui.addon.onLoad(h)` | `data = { addon }` |
| `panel:player-detail:edit-modal:load` | `pano.ui.player.onEditLoad(h)` | `data = { player }` |

## 7. Поверхности авторизации (только тема)

Помощники для страниц авторизации. `<page>` — это одна из `login`, `register`, `resetPassword`, `activate`, `activateNewEmail`, `renewPassword`.

| Вызов | Назначение |
|---|---|
| `pano.ui.auth.<page>.content.edit(callback)` / `.get()` | Отредактировать / прочитать слот контента этой страницы. Это тот же слот, что и `<page>-content` из §4 (например, `pano.ui.auth.login.content.edit` редактирует слот `login-content`) — сокращение к нему, а не отдельный механизм. |
| `pano.ui.auth.<page>.onLoad(handler)` | Подписаться на событие загрузки этой страницы (события `theme:<page>:load` из §6). |
| `pano.ui.auth.login.alternativeMethods.add(method)` / `.get()` | Добавить / прочитать альтернативный метод входа (например, кнопку social-login). |
| `pano.ui.auth.register.alternativeMethods.add(method)` / `.get()` | То же для регистрации. |
| `pano.ui.auth.login.load(event)` | Запустить поток загрузки входа (для страницы плагина, которая представляет собственный вход). Возвращает `{ error, username, event }`. |
| `pano.ui.auth.register.load(event)` | То же для страницы регистрации плагина. |
| `pano.ui.auth.login.form.get()` | Возвращает собственный компонент тела формы входа темы, чтобы страница плагина могла отрисовать стандартную форму входа внутри собственного макета. |
| `pano.ui.auth.register.form.get()` | То же, для формы регистрации. |

`resetPassword`, `activate`, `activateNewEmail` и `renewPassword` предоставляют только `content.edit`/`content.get` и `onLoad`.

| `@panomc/sdk/components/theme` | `PlayerHead`, `NoContent`, `Date`, `Toast`, `PageTitle`, `PageActions`, `Pagination` |
| `@panomc/sdk/components/panel` | `NoContent`, `Editor`, `DragAndDropZone`, `Date`, `Toast`, `PageLoading`, `PageActions`, `PageLoader`, `PageNavItem`, `PageNav`, `Pagination`, `CardFilters`, `CardFiltersItem`, `CardHeader`, `SearchInput` |
| `@panomc/sdk/variables` | `API_URL`, `UI_URL`, `PANEL_URL`, `SETUP_URL`, `PANO_WEBSITE_URL`, `PANO_WEBSITE_API_URL`, `PRERELEASE`, `COOKIE_PREFIX`, `CSRF_TOKEN_COOKIE_NAME`, `JWT_COOKIE_NAME`, `CSRF_HEADER`, `updateApiUrl`, `updatePanoWebsiteUrl`, `updatePanoWebsiteApiUrl` |
| `@panomc/sdk/svelte` | `page`, `base`, `navigating`, `browser`, `goto`, `invalidate`, `invalidateAll`, `error`, `redirect` |
| `@panomc/sdk/internal` | `setPanoContext`, `getPanoContext` |

`viewComponent` появляется дважды — в `@panomc/sdk` и `@panomc/sdk/utils/component`. Это **та же функция**; импортируйте из любого (`@panomc/sdk` — обычный путь).

Экспорты, чьи имена не самоочевидны:

- `_` — функция/хранилище перевода: `$_('some.key')` в компоненте. См. [Локализацию](/ru/addon/localization/).
- `languageLoading`, `currentLanguage` — хранилища для текущего состояния i18n.
- `tooltip` — Svelte-action: прикрепите его через `use:tooltip` на элементе.
- `copy` — копирует строку в буфер обмена.
- `hasPermission(permission, user)` — возвращает `true`, если `user` держит `permission`; используйте, чтобы показывать/скрывать UI (см. §4).
- `showToast` / `limitTitle` — всплывающее уведомление / обрезание длинной строки заголовка для него (см. сигнатуру `showToast` ниже).
- `NoContent` — компонент-заглушка «пустого состояния».
- `PageLoading` / `PageLoader` — UI состояния загрузки (два варианта; вставьте каждый, чтобы увидеть, какой подходит вашему случаю).
- `PlayerHead` — отрисовывает изображение головы игрока Minecraft.

::: tip Какие `@panomc/sdk/variables` вам на самом деле нужны
Повседневное использование: `API_URL`, `UI_URL`, `PANEL_URL` (и `PRERELEASE`, чтобы определить предрелизную сборку). Остальные — `CSRF_TOKEN_COOKIE_NAME`, `JWT_COOKIE_NAME`, `CSRF_HEADER`, `COOKIE_PREFIX`, сеттеры `update*Url` — это «водопровод» авторизации/безопасности, который `ApiUtil` уже обрабатывает за вас; вы их редко трогаете.
:::

::: tip `@panomc/sdk/svelte` = собственные API SvelteKit
Они зеркалят экспорты SvelteKit — `page`, `navigating`, `browser` (из `$app/state` / `$app/environment`), `base` (из `$app/paths`), `goto`, `invalidate`, `invalidateAll` (из `$app/navigation`) и `error`, `redirect` (из `@sveltejs/kit`). Как ведёт себя каждый — см. в документации SvelteKit. Импортируйте их из `@panomc/sdk/svelte`, а не из `$app/...`, чтобы они разрешались в среду выполнения хоста.
:::

::: warning Нет `Button`, `Card` или `Input`
`@panomc/sdk/components/panel` и `.../theme` экспортируют ровно те компоненты, что перечислены выше. Нет общих `Button`/`Card`/`Input` — некоторые ранние примеры ссылались на компоненты, которых никогда не существовало; список выше — авторитетный. Стройте простые элементы управления обычной разметкой или переиспользуйте перечисленные компоненты.
:::

**Сигнатуры методов `ApiUtil`** (все `async`, все принимают один объект опций):

| Метод | Опции |
|---|---|
| `ApiUtil.get(...)` | `{ path, request, csrfToken, token, blob, handler }` |
| `ApiUtil.post(...)` | `{ path, request, body, headers, csrfToken, token, blob, handler, onUploadProgress }` |
| `ApiUtil.put(...)` | `{ path, request, body, headers, csrfToken, token, blob, handler, onUploadProgress }` |
| `ApiUtil.delete(...)` | `{ path, request, headers, csrfToken, token, blob, handler }` |
| `ApiUtil.customRequest(...)` | `{ path, data, request, csrfToken, token, blob, handler, onUploadProgress }` |

**Что означает каждая опция** (не каждая опция появляется в каждом методе — используйте таблицу сигнатур выше):

| Опция | Значение |
|---|---|
| `path` | Путь API, **относительно `/api`** — передайте `'shoutbox/list'`, и утилита вызовет `/api/shoutbox/list`. |
| `request` | Аргумент `load(event)`. Передавайте его всякий раз, когда вызываете изнутри `load()`, чтобы у запроса был CSRF-токен и он работал во время SSR (см. заметку под примером). |
| `body` | Нагрузка запроса (объект, отправляемый как JSON; или `FormData` для загрузки файлов). Только POST/PUT. |
| `headers` | Дополнительные заголовки запроса. POST/PUT/DELETE. |
| `csrfToken` | CSRF-токен. Обычно вы его опускаете — утилита читает его из сессии через `request`. |
| `token` | bearer-токен; когда установлен, отправляется как `Authorization: Bearer <token>`. Опустите для обычных вызовов вошедшего пользователя — авторизацию обрабатывают куки. |
| `blob` | Установите `true`, когда ответ — файл/бинарные данные, чтобы он читался как Blob, а не разбирался как JSON. |
| `handler` | Необязательный колбэк `(data, reject) => data`, который постобрабатывает разобранный ответ до того, как он вернётся вам. |
| `onUploadProgress` | Колбэк прогресса загрузки (POST/PUT/customRequest) — используйте его, чтобы двигать индикатор прогресса. |
| `data` | (только customRequest) сырые опции fetch — `method`, `body`, `headers`. Помощники `get`/`post`/и т. д. строят это за вас. |

Минимальный GET во время загрузки страницы — обратите внимание на `path` (относительно `/api`) и `request: event` на позиции:

```js
import ApiUtil from '@panomc/sdk/utils/api';

export async function load(event) {
  // pass request: event so the call works during SSR (the first page view)
  const response = await ApiUtil.get({ path: 'your-endpoint', request: event });
  return { response }; // this object becomes your page component's props
}
```

Всегда передавайте `request: event`, когда вызываете их внутри `load()`. Если забудете, вызов не сможет подхватить CSRF-токен или переиспользовать серверный fetch во время SSR — он может провалиться или молча перезапуститься в браузере после загрузки страницы. Последний случай — самый сбивающий с толку: выглядит, будто всё работает, когда вы кликаете по сайту, но ломается при свежей загрузке страницы или жёстком обновлении.

**Сигнатура `showToast`:** `showToast(text, params = {}, toastComponent)`.

| Аргумент | Значение |
|---|---|
| `text` | Сообщение. Если это ключ перевода (содержит `.`), оно переводится; иначе показывается как есть. |
| `params` | С toast по умолчанию они становятся **значениями** перевода, интерполируемыми в `text`. С кастомным `toastComponent` они передаются этому компоненту как его props. Необязательно (по умолчанию `{}`). |
| `toastComponent` | Необязательный кастомный Svelte-компонент для отрисовки вместо toast по умолчанию. |

## 10. Что вам можно импортировать

::: warning Только замороженный список разрешается в среду выполнения хоста
Ваша сборка намеренно **не** встраивает эти импорты в JS вашего аддона — она оставляет их **внешними**, и хост поставляет их во время выполнения, чтобы каждый аддон разделял один экземпляр Svelte. Импортируйте что-либо вне списка, и оно не разрешится во время выполнения.
:::

Разрешённые голые спецификаторы — ровно эти:

- Каждый спецификатор `@panomc/sdk` из таблицы §9.
- Svelte: `svelte`, `svelte/store`, `svelte/transition`, `svelte/easing`, `svelte/motion`, `svelte/animate`, `svelte/legacy`, `svelte/events`, `svelte/attachments`, `svelte/reactivity`, `svelte/reactivity/window` и `svelte-i18n`.
- **Фиксированный** набор внутренностей Svelte *(продвинутое — компилятор эмитит эти импорты в ваш собранный код за вас; вы никогда не пишете их вручную)*: `svelte/internal`, `svelte/internal/client`, `svelte/internal/disclose-version`, `svelte/internal/flags/legacy`, `svelte/internal/flags/async` и `svelte/internal/flags/tracing`. Это точный список, **а не** wildcard `svelte/internal/*`; любой другой подпуть `svelte/internal/...` не разрешается.

Всё остальное — `chart.js`, `svelte-select`, любой другой npm-пакет — должно быть **встроено в ваш аддон** вашей сборкой rollup, а не импортировано голым. Вы не делаете для этого ничего особенного: сборка rollup из boilerplate уже встраивает всё, что вы `bun add`, а затем `import` обычным образом. (Аддон market делает именно это, чтобы поставлять Chart.js.)

::: warning Никогда не добавляйте `svelte` в свой `package.json`
SDK контролирует, с какой версией Svelte все компилируют (его версия закреплена), и сборка **проваливается** при несовпадении. Вторая копия `svelte` в вашем `package.json` молча ломает ваши страницы (две копии не согласуются во время гидрации). См. [Архитектуру](/ru/addon/architecture/).
:::

## Известные мёртвые поверхности (не используйте)

Для полноты: два члена существуют на поверхности, но ничего не делают — не стройте на них:

- `pano.debug` — булев флаг на объекте `pano`. Сейчас он жёстко зашит в `false`, и ни один хост его не устанавливает, поэтому **не** используйте его для определения dev-сборки.
- `onContextUpdate()` — старый метод boilerplate, который ни один хост не вызывает (см. §1).
