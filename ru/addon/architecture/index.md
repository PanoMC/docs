# Архитектура

Вы собрали и загрузили аддон в [Начале работы](/ru/addon/getting-started/). Эта страница — карта. Она объясняет, что на самом деле произошло: что находится внутри того единственного jar (jar — это просто zip со скомпилированным кодом и ресурсами, вы можете открыть его любым архиватором), что Pano делает, когда загружает его, и где оказывается каждый написанный вами файл, пока аддон работает.

К концу вы сможете объяснить весь жизненный цикл загрузки, поймёте, **почему нельзя добавлять `svelte` в зависимости**, и сможете показать, где живёт каждый файл репозитория, когда аддон работает. По пути врезки «Убедитесь сами» дают вам что-то распаковать, открыть или кликнуть, чтобы идеи были не просто словами.

## Один jar, три среды выполнения

Аддон поставляется как **единственный jar**, но код внутри него выполняется в трёх разных местах. Вот вся картина с одного взгляда:

```text
                        ┌──────────────────────┐
                        │  your addon = 1 jar   │
                        └───────────┬──────────┘
          ┌─────────────────────────┼─────────────────────────┐
          ▼                         ▼                         ▼
     Browser                  Node (theme + panel)       JVM (Pano server)
     client bundle            server bundle              Kotlin backend
     the UI users click       renders your UI to HTML    endpoints, DB, events
```

1. **Бэкенд на Kotlin** — загружается [PF4J](https://pf4j.org/), загрузчиком плагинов, который использует Pano: он находит ваш jar в папке `plugins` и загружает его классы в работающий сервер Pano (JVM). Это ваши эндпоинты, таблицы базы данных, права доступа и слушатели событий. Он выполняется в том же процессе, что и сам Pano.
2. **Клиентский *client*-бандл на Svelte** — «бандл» здесь просто означает ваши `.svelte`-файлы, скомпилированные в один `.mjs`-файл. Он выполняется в браузере посетителя, внутри **и** панели, **и** активной темы. Это UI, который пользователь фактически кликает.
3. **Серверный *server*-бандл на Svelte** — тот же UI, скомпилированный так, чтобы тема и панель (которые являются процессами Node) могли отрисовать ваши компоненты в HTML ещё до того, как браузер их увидит, — это серверный рендеринг (SSR). Затем браузер «гидрирует» этот HTML: он повторно выполняет те же компоненты и присоединяется к разметке, которая уже есть на странице. Чтобы гидрация работала, **обе стороны должны выполнять один и тот же Svelte** — держите это в уме, это причина большого правила ближе к концу страницы.

Важная часть: существует **один** файл-точка входа UI, `src/main.js`, и он обслуживает панель *и* тему. Вы не пишете два UI — вы пишете один и ветвитесь по тому, где он выполняется.

Это метод класса, который вы экспортируете по умолчанию из `src/main.js` (тот же класс, что вы видели в Начале работы). Pano вызывает его за вас, когда загружается UI, — см. [Что происходит, когда загружается UI](#что-происходит-когда-загружается-ui) ниже — и передаёт ему объект `pano` с флагом `isPanel`:

```js
onLoad() {
  const { pano } = this;
  if (pano.isPanel) {
    // panel registrations
  } else {
    // theme registrations
  }
}
```

Итак, jar несёт бэкенд, выполняющийся в JVM Pano, и UI, выполняющийся в двух контекстах Node/браузера, — и всё это собрано из одной папки исходников. Остальная часть страницы прослеживает каждый из них до того места, где он живёт во время выполнения.

## Где оказывается каждый файл

Вот репозиторий аддона Shoutbox с пометкой о том, чем каждая часть становится во время выполнения. Имена папок — это соглашения, но разделение между `src/` (UI) и `src/main/` (бэкенд + ресурсы) фиксировано.

```text
pano-plugin-shoutbox/
├─ build.gradle.kts             # Gradle build
├─ gradle.properties            # the addon's metadata (explained below)
├─ package.json                 # UI dependencies (never lists svelte — see below)
├─ rollup.config.js             # builds the Svelte UI
├─ src/
│  ├─ main.js                   # the single UI entry (panel + theme)
│  ├─ panel/                    # Svelte components shown in the panel
│  │  └─ ShoutboxSettings.svelte
│  ├─ theme/                    # Svelte components shown in the theme
│  │  └─ ShoutboxWidget.svelte
│  └─ main/
│     ├─ kotlin/com/panomc/plugins/shoutbox/
│     │  ├─ ShoutboxPlugin.kt   # your PanoPlugin subclass — the entry point
│     │  ├─ config/             # ShoutboxConfig
│     │  ├─ db/
│     │  │  ├─ dao/             # ShoutDao (abstract): one class per table, declares its queries
│     │  │  ├─ impl/            # ShoutDaoImpl — carries the @Dao annotation, runs the dao's queries
│     │  │  ├─ model/           # Shout entity
│     │  │  └─ migration/       # DatabaseMigration classes (see note under the tree)
│     │  ├─ routes/
│     │  │  ├─ api/             # public endpoints (GetShoutsAPI)
│     │  │  └─ panel/           # panel endpoints (PanelApi)
│     │  ├─ event/              # event listeners
│     │  ├─ permission/         # ManageShoutboxPermission
│     │  └─ log/                # CreatedShoutLog
│     └─ resources/
│        ├─ config.conf         # default config
│        ├─ logo.png            # addon icon shown in the panel
│        ├─ locales/            # en-US.json, tr.json, ru.json
│        └─ plugin-ui/          # generated by the build — never edit by hand (gitignored)
```

Два имени папок выше заслуживают пояснения простыми словами:

- Разделение между `dao/` и `impl/`: класс `dao/` **абстрактный** и лишь *объявляет* каждый запрос к базе данных; соответствующий класс `impl/` — конкретный, это тот класс, который несёт аннотацию `@Dao` и содержит код, который на самом деле *выполняет* запросы.
- `migration/` — миграция это небольшой класс, который обновляет таблицы или конфигурацию уже существующего экземпляра, когда новая версия вашего аддона меняет их форму (например, добавляет столбец). Она выполняется один раз, только на тех экземплярах, что отстали.

Два правила, по которым читать остальную часть дерева:

- Всё под `src/main/kotlin` и `src/main/resources` — это **содержимое бэкенд-jar**. Kotlin компилируется в классы; ресурсы (конфигурация, изображения, локали) копируются без изменений; и то, и другое попадает в jar.
- Всё под `src/`, но *вне* `src/main/` (`main.js`, `panel/`, `theme/`) — это **исходники UI**. Когда вы собираете, компилятор превращает их в файлы, которые попадают внутрь ваших бэкенд-ресурсов:

| Исходник UI (вы пишете) | Скомпилированный вывод (пишет сборка) |
|---|---|
| `src/main.js`, `src/panel/`, `src/theme/` | `src/main/resources/plugin-ui/` |

Да — сборка пишет свой вывод *обратно в ваше дерево исходников*, внутрь `src/main/resources/`. Это сделано намеренно: помещение скомпилированного UI под `resources/` — это ровно то, что заставляет его быть заархивированным и отправленным внутри того же jar. (Это ещё и причина, по которой `plugin-ui/` в gitignore — он регенерируется при каждой сборке, поэтому коммитить нечего.)

::: tip Каждая папка здесь необязательна, кроме класса-точки входа
`config/`, `db/`, `event/`, `permission/`, `log/` — это просто места, куда соглашение помещает вещи. Pano находит ваши классы по их **аннотациям** — аннотация это тег `@Что-то`, написанный над классом, и Pano сканирует ваши скомпилированные классы в поисках этих тегов, — а не по их папке (об этом далее). У аддона, состоящего только из конфигурации, вроде `pano-plugin-cookies` почти нет этих папок; у полноценного CRUD-аддона (create/read/update/delete — то есть у него есть таблицы базы данных *и* эндпоинты) вроде `pano-plugin-announcement` есть все они.
:::

::: tip Убедитесь сами
Скопируйте свой собранный `*.jar` и переименуйте копию в `*.zip`, затем откройте её (jar — это просто zip). Внутри вы должны найти `plugin-ui.zip`, лежащий рядом с папкой `locales/` и `logo.png` — этот единственный `plugin-ui.zip` и есть весь ваш скомпилированный UI, упакованный внутрь.
:::

## Что происходит, когда загружается бэкенд

Если вы раньше писали плагины для Minecraft, обратите внимание: здесь **нет `plugin.yml`**. В любом случае все метаданные — id, главный класс, требуемая версия Pano — живут в `MANIFEST.MF` jar-файла (небольшой текстовый файл внутри каждого jar, который его описывает), записываемом за вас из `gradle.properties` во время сборки. См. [Конфигурация манифеста](/ru/addon/manifest/) о том, какой именно ключ отображается в какой атрибут.

::: tip Убедитесь сами
Снова откройте собранный jar как zip и прочитайте `META-INF/MANIFEST.MF` в текстовом редакторе — вы должны увидеть внутри id и главный класс вашего аддона.
:::

Когда Pano загружает ваш jar, PF4J находит главный класс, названный в манифесте, создаёт его экземпляр и вызывает ваши хуки жизненного цикла **по порядку**:

```text
jar load → onCreate() → onEnable() → onStart() → … running … → onStop() → onDisable() → onUninstall()
```

- Все хуки — это `suspend`-функции (`suspend` — это метка Kotlin для асинхронности; на практике это означает, что вы можете вызывать функции базы данных и сети Pano прямо внутри этих хуков), и все они по умолчанию ничего не делают — вы переопределяете только те, что вам нужны.
- `onStart()` — это место, где большинство аддонов выполняют свою настройку (инициализируют базу данных, загружают конфигурацию). Руководство [Разработка бэкенда](/ru/addon/backend/) проходит канонический паттерн.
- `onUninstall()` вызывается только тогда, когда владелец сайта **удаляет** аддон в панели, — а не когда он просто его отключает. Здесь `pano-plugin-shoutbox` удалил бы свою таблицу `shout` из базы данных.

::: tip Убедитесь сами
Поместите строку `logger.info(...)` внутрь `onStart()`, пересоберите и перезагрузите аддон — строка появится в консоли Pano. Это самый простой способ увидеть, как жизненный цикл действительно срабатывает.
:::

### Ваши классы находятся автоматически

Когда ваш аддон загружается, Pano проходит по каждому классу в вашем пакете — `com.panomc.plugins.shoutbox` и всём, что ниже. Любой класс, несущий один из тегов в таблице ниже (аннотации `@Что-то`), создаётся за вас автоматически; вы никогда не конструируете их сами.

Более того, Pano заполняет то, что нужно каждому классу. Если вашему эндпоинту нужен `ShoutDao` для чтения базы данных, вы просто перечисляете `ShoutDao` как параметр конструктора, и Pano передаёт готовый экземпляр — вы никогда не пишете `ShoutDaoImpl()` сами. Передача классу того, от чего он зависит, вместо того чтобы заставлять его строить это, называется **внедрением зависимостей** (dependency injection).

| Аннотация | Что она регистрирует |
|---|---|
| `@Endpoint` | HTTP-маршрут — оживает в тот же миг, как загружается аддон |
| `@Dao` | объект доступа к данным для одной из ваших таблиц — ставится на конкретный класс *impl*, а не на абстрактный DAO |
| `@Migration` | выполняется один раз на экземпляр, чтобы привести старую базу данных или конфигурацию к вашей новой версии |
| `@EventListener` | слушатель событий платформы (завершена установка, удалён игрок, …) |
| `@PermissionDefinition` | узел права доступа, который панель может выдавать |

Вы никогда не вызываете метод «зарегистрируй этот эндпоинт» — аннотирование класса *и есть* регистрация. Маршруты, объявленные классами `@Endpoint`, поднимаются, когда аддон загружается, и снова удаляются, когда он выгружается, — так что включение и отключение аддона чисто добавляет и удаляет его API.

Фреймворк, делающий всё это, — **Spring**. Каждый аддон получает свой собственный *контекст приложения* Spring (изолированный контейнер, который держит объекты, созданные Spring), и Spring *сканирует компоненты* вашего пакета — это сканирование и есть шаг «пройти по каждому классу» выше. Слово Spring для объекта, который он создаёт и которым управляет, — **bean**, так что «ваши bean-ы находятся автоматически» — это просто заумный способ сказать «ваши аннотированные классы создаются за вас».

::: tip Как дотянуться до собственных сервисов Pano
**Коротко:** ваши собственные классы появляются в конструкторах автоматически; чтобы использовать один из встроенных сервисов Pano, вы получаете его одной строкой ниже. К таким сервисам относятся `DatabaseManager`, `AuthProvider`, `SetupManager`, `PluginDatabaseManager` и другие — вы обращаетесь к ним, когда вашему коду нужно что-то, чем владеет сам Pano (например, спросить у `SetupManager`, завершена ли первичная установка).

Две вещи о строке ниже: `applicationContext` унаследован от `PanoPlugin` (ваш класс-точка входа расширяет его), а `by lazy` означает, что bean извлекается в первый момент, когда вы фактически используете `setupManager`, а не при запуске. (`SetupManager::class.java` — это просто способ Kotlin назвать нужный вам тип.)

```kotlin
private val setupManager by lazy {
    applicationContext.getBean(SetupManager::class.java)
}
```

К любому сервису хоста вы дотягиваетесь этим способом — `applicationContext.getBean(...)` с типом сервиса.

**Продвинутое, если вам интересно, зачем нужно это извлечение:** bean-ы вашего аддона живут в контексте под названием `pluginBeanContext`, тогда как сервисы Pano живут в *отдельном* контексте хоста (`applicationContext`). Сканирование компонентов видит только ваш пакет, поэтому сервисы хоста **не** внедряются в ваши конструкторы — вы должны запросить их явно, как выше.
:::

## Что происходит, когда загружается UI

UI никогда не поставляется как отдельные файлы. Когда вы запускаете релизную сборку, скомпилированные папки `plugin-ui/{client,server}` архивируются в единственный ресурс jar, **`plugin-ui.zip`**. Оттуда:

1. При загрузке Pano вычисляет **хеш UI** этого zip и объявляет `{ version, uiHash }` для вашего аддона через API информации о сайте (`/api/siteInfo`), который тема и панель уже вызывают.
2. Тема (браузер) импортирует ваш **client**-бандл с параметром для сброса кеша — `client.mjs?v=<uiHash>` — так что новая сборка инвалидирует старую закешированную копию. Процесс Node импортирует **server**-бандл, `server/server.mjs`, для SSR.
3. После импорта Pano конструирует ваш экспортированный по умолчанию класс и вызывает его `onLoad()` — тот же `onLoad()` из начала этой страницы. Это единственная точка входа, где вы регистрируете всё, что добавляет UI: хуки, страницы, ссылки навигации.

::: tip Убедитесь сами
Откройте инструменты разработчика браузера, перейдите на вкладку **Network** и загрузите страницу, которая использует ваш аддон. Найдите запрос к `client.mjs?v=<hash>` — этот `<hash>` и есть хеш UI, который Pano только что передал браузеру.
:::

Хеш UI важен в основном для **выпущенных** аддонов, и в разработке он ведёт себя иначе:

- **Выпущенный:** новая версия поставляет новый zip, поэтому хеш меняется, и параметр для сброса кеша подтягивает свежий бандл.
- **Режим разработки:** при включённом [Режиме разработки](/ru/addon/getting-started/) Pano пересобирает ваш zip UI с диска на каждый запрос. Тогда хеш не является настоящим хешем содержимого — это фиксированное значение-заглушка, `dev-build`, — и тема заново запрашивает бандл на каждый запрос вместо кеширования. Именно это заставляет изменение при `bun run dev` появляться по F5.

Руководство [Разработка фронтенда](/ru/addon/frontend/) охватывает то, что вы делаете внутри `onLoad()`.

## Общая среда выполнения Svelte

Это самая важная вещь, которую нужно понять об UI, и она объясняет правило, которое всех удивляет.

Ваш client-бандл **не содержит Svelte, `svelte-i18n` или `@panomc/sdk`.** Сборка rollup намеренно оставляет эти импорты *внешними*: собранный файл всё ещё буквально говорит `import ... from 'svelte'`, и ничего не было скопировано внутрь — так что что-то во время выполнения должно ответить на этот импорт.

Это «что-то» — хост (тема или панель). Он предоставляет **import map** — небольшой кусочек JSON, который браузер читает, чтобы узнать, откуда голое имя вроде `'svelte'` должно на самом деле загружаться, — и эта карта направляет каждый из этих импортов на URL `/runtime/...`. Каждый URL `/runtime` просто реэкспортирует собственную живую копию модуля хоста.

В результате Pano и все его аддоны разделяют **одну** среду выполнения Svelte и **один** экземпляр SDK. На практике это означает, что ваши компоненты и компоненты темы разделяют состояние и перерисовываются вместе, как будто они всегда были одним приложением. Если бы ваш аддон включал собственную приватную копию Svelte или SDK, у этой копии было бы своё отдельное состояние, и гидрация (из начала этой страницы — повторное присоединение браузера к серверно-отрисованному HTML) сломалась бы.

::: tip Убедитесь сами
Откройте любую страницу темы и используйте **Просмотр исходного кода** (сырой HTML до того, как выполнится JavaScript). Если ваш аддон что-то отрисовывает на этой странице, вы увидите его разметку уже сидящей там, в HTML, — это серверно-отрисованный вывод, который браузер собирается гидрировать.
:::

Есть жёсткое следствие: поскольку скомпилированный вывод Svelte гарантированно работает только с **точно той же** версией Svelte, на которой он будет выполняться, версия компилятора вашей сборки должна в точности совпадать с версией хоста. Именно поэтому вы никогда не закрепляете Svelte сами — `@panomc/sdk` закрепляет правильную версию, а ваш `rollup.config.js` отказывается собирать при несовпадении.

::: warning Никогда не добавляйте `svelte` в свой `package.json`
Версия Svelte приходит из закрепления в `@panomc/sdk`, а не от вас. Если вы перечислите `svelte` сами, bun может установить версию, отличную от той, что закрепил `@panomc/sdk`, и сборка защищается ровно от этого: при несовпадении версий `rollup.config.js` печатает ошибку, и сборка проваливается (код выхода 1) — намеренно. Расхождение версий ломает гидрацию способами, которые больно отлаживать, поэтому защита останавливает вас до отправки. Удалите любую запись `svelte` и переустановите.
:::

Ещё одну вещь это объясняет: *единственные* голые импорты, остающиеся внешними, — это Svelte, `svelte-i18n` и `@panomc/sdk` (и их подпути). «Голый» импорт — это импорт, записанный по имени пакета — `import x from 'chart.js'` — в отличие от пути вроде `./file.js`. Всё остальное, что вы импортируете, — сторонний пакет вроде библиотеки графиков — не имеет шима хоста, поэтому **должно быть встроено** в вашу сборку. Эта часть автоматическая: просто `bun add` его и импортируйте, и сборка сама скопирует его внутрь — особыми являются только те три общих имени. [Справочник API фронтенда](/ru/addon/api-reference/) перечисляет точный набор разрешённых голых спецификаторов.

## Где живут ваши данные во время выполнения

Во время выполнения ваш аддон трогает два очень разных места хранения, и полезно держать их раздельно:

| Расположение | Что оно хранит | Кто пишет в него |
|---|---|---|
| `plugins/<pluginId>/` (каталог данных) | `config.conf`, загруженные файлы, всё, что ваш аддон сохраняет на диск | создаётся автоматически при первой загрузке; пишете его вы (например, через `PluginConfigManager`, показан в Разработке бэкенда) |
| Внутри jar (ресурсы) | `locales/*.json`, `logo.png`, `plugin-ui.zip` | «запекаются» во время сборки — **только для чтения** во время выполнения |

Каталог данных назван по вашему `pluginId` (для Shoutbox — `plugins/pano-plugin-shoutbox/`) и переживает перезапуски — это место, где живёт состояние конкретной установки. В **выпущенном** jar эти ресурсы фиксированы в тот момент, когда jar собран. Однако во время **разработки** Pano обслуживает их с диска в реальном времени: при включённом Режиме разработки он читает `locales/*.json` прямо с диска и пересобирает ваш zip UI с диска на каждый запрос — что и заставляет работать цикл «правь-и-обновляй». Код Kotlin — единственное, что никогда не перезагружается на лету: изменения в Kotlin требуют пересборки и перезапуска Pano. (В Начале работы есть полная таблица «горячее против пересборки».)

## Что дальше

Теперь, когда у вас есть ментальная модель, выберите сторону для сборки:

- **[Разработка бэкенда](/ru/addon/backend/)** — добавьте таблицу, откройте API, защитите его правом доступа, залогируйте действие.
- **[Разработка фронтенда](/ru/addon/frontend/)** — смонтируйте компонент на главной странице, добавьте страницу панели, вызовите свой API.
- **[Конфигурация манифеста](/ru/addon/manifest/)** — ключи `gradle.properties`, которые становятся манифестом вашего jar.

Хотите почитать настоящий код аддона? Два примера, упомянутых выше, — это `pano-plugin-cookies` (только конфигурация, почти без бэкенд-папок) и `pano-plugin-announcement` (полноценный CRUD-аддон, использующий их все).
