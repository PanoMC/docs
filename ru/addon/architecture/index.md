# Архитектура

Вы собрали и загрузили аддон в разделе [Начало работы](/ru/addon/getting-started/). Эта страница объясняет, что на самом деле произошло: что находится внутри того единственного jar, что делает Pano, когда загружает его, и где оказывается каждый написанный вами файл во время работы.

К концу вы сможете объяснить весь жизненный цикл загрузки, будете знать, **почему нельзя добавлять `svelte` в зависимости**, и сможете указать, где в репозитории живёт каждый файл, когда аддон запущен.

## Один jar, три среды выполнения

Аддон поставляется как **единственный jar**, но код внутри него выполняется в трёх разных местах:

1. **Бэкенд на Kotlin** — загружается [PF4J](https://pf4j.org/) прямо в JVM Pano. Это ваши эндпоинты, таблицы базы данных, права доступа и слушатели событий. Он выполняется в том же процессе, что и сам Pano.
2. **Клиентский бандл Svelte** — выполняется в браузере посетителя, **как** в панели, **так и** в активной теме. Это тот UI, по которому пользователь фактически кликает.
3. **Серверный бандл Svelte** — тот же UI, скомпилированный для **серверного рендеринга (SSR)**. Тема и панель — это Node-процессы, которые рендерят ваши компоненты в HTML до того, как их увидит браузер.

Важная деталь: есть **один** файл-точка входа UI, `src/main.js`, и он обслуживает *и* панель, *и* тему. Вы пишете не два UI — вы пишете один и ветвитесь по тому, где он выполняется:

```js
onLoad() {
  const { pano } = this;
  if (pano.isPanel) {
    // регистрации для панели
  } else {
    // регистрации для темы
  }
}
```

Итак, jar несёт бэкенд, выполняющийся в JVM Pano, и UI, выполняющийся в двух контекстах Node/браузера, — и всё это собрано из одного дерева исходников. Остальная часть страницы прослеживает каждый из них до места, где он живёт во время работы.

## Где оказывается каждый файл

Вот репозиторий аддона Shoutbox с пометкой о том, чем становится каждая часть во время работы. Сравните его с деревом, которое поставляет boilerplate — имена папок являются соглашениями, но разделение между `src/` (UI) и `src/main/` (бэкенд + ресурсы) фиксировано.

```text
pano-plugin-shoutbox/
├─ build.gradle.kts             # сборка Gradle
├─ gradle.properties            # манифест — id, class, versions (см. страницу Манифест)
├─ package.json                 # зависимости UI (никогда не перечисляет svelte — см. ниже)
├─ rollup.config.js             # собирает UI на Svelte
├─ src/
│  ├─ main.js                   # единственная точка входа UI (панель + тема)
│  ├─ panel/                    # Svelte-компоненты, показываемые в панели
│  │  └─ ShoutboxSettings.svelte
│  ├─ theme/                    # Svelte-компоненты, показываемые в теме
│  │  └─ ShoutboxWidget.svelte
│  └─ main/
│     ├─ kotlin/com/panomc/plugins/shoutbox/
│     │  ├─ ShoutboxPlugin.kt   # ваш подкласс PanoPlugin — точка входа
│     │  ├─ config/             # ShoutboxConfig
│     │  ├─ db/
│     │  │  ├─ dao/             # ShoutDao (abstract)
│     │  │  ├─ impl/            # ShoutDaoImpl (@Dao)
│     │  │  ├─ model/           # сущность Shout
│     │  │  └─ migration/       # классы DatabaseMigration
│     │  ├─ routes/
│     │  │  ├─ api/             # публичные эндпоинты (GetShoutsAPI)
│     │  │  └─ panel/           # эндпоинты панели (PanelApi)
│     │  ├─ event/              # слушатели событий
│     │  ├─ permission/         # ManageShoutboxPermission
│     │  └─ log/                # CreatedShoutLog
│     └─ resources/
│        ├─ config.conf         # конфигурация по умолчанию
│        ├─ logo.png            # иконка аддона, показываемая в панели
│        ├─ locales/            # en-US.json, tr.json, ru.json
│        └─ plugin-ui/          # собранный UI — упаковывается в plugin-ui.zip при сборке
```

Два правила для чтения этого дерева:

- Всё, что находится под `src/main/kotlin` и `src/main/resources`, — это **содержимое бэкенд-jar**. Оно компилируется и упаковывается в jar как есть.
- Всё, что находится под `src/`, но *вне* `src/main/` (`main.js`, `panel/`, `theme/`), — это **исходники UI**. Сборка компилирует их в `src/main/resources/plugin-ui/`, который затем упаковывается и поставляется внутри того же jar.

::: tip Здесь каждая папка необязательна, кроме класса-точки входа
`config/`, `db/`, `event/`, `permission/`, `log/` — это просто места, куда соглашение кладёт вещи; Pano находит ваши классы по их аннотациям, а не по папке (об этом далее). У аддона только с конфигурацией, вроде `pano-plugin-cookies`, почти ничего из этого нет; у CRUD-аддона вроде `pano-plugin-announcement` есть всё.
:::

## Что происходит, когда загружается бэкенд

У Pano **нет `plugin.yml`**. Все метаданные — id, главный класс, требуемая версия Pano — живут в `MANIFEST.MF` jar-файла, записанном за вас из `gradle.properties` при сборке. См. [Конфигурацию манифеста](/ru/addon/manifest/), чтобы узнать точно, какой ключ соответствует какому атрибуту.

Когда Pano загружает ваш jar, PF4J находит главный класс, указанный в манифесте, создаёт его экземпляр и вызывает ваши хуки жизненного цикла **по порядку**:

```text
jar load → onCreate() → onEnable() → onStart() → … running … → onStop() → onDisable() → onUninstall()
```

- Все хуки — это `suspend`-функции, и все по умолчанию **ничего не делают** (no-op) — вы переопределяете только те, что вам нужны.
- `onStart()` — это место, где большинство аддонов выполняют настройку (инициализируют базу данных, загружают конфигурацию). Руководство [Разработка бэкенда](/ru/addon/backend/) проходит канонический шаблон.
- `onUninstall()` вызывается только тогда, когда владелец сайта **удаляет** аддон в панели — а не когда он просто отключает его. Здесь `pano-plugin-shoutbox` удалил бы свою таблицу `shout`.

### Ваши бины находятся автоматически

В момент загрузки вашего аддона Pano даёт ему **собственный контекст приложения Spring** и сканирует компоненты **только в вашем поддереве пакетов** (`com.panomc.plugins.shoutbox` и ниже). Любой класс, несущий одну из этих аннотаций, создаётся за вас, с внедрением зависимостей его конструктора:

| Аннотация | Что она регистрирует |
|---|---|
| `@Endpoint` | HTTP-маршрут — становится активным в тот же миг, когда загружается аддон |
| `@Dao` | объект доступа к базе данных для одной из ваших таблиц |
| `@Migration` | миграцию схемы или конфигурации |
| `@EventListener` | слушателя событий платформы (завершение установки, удаление игрока, …) |
| `@PermissionDefinition` | узел права доступа, который панель может выдавать |

Вы никогда не вызываете метод «зарегистрировать этот эндпоинт» — аннотирование класса и есть регистрация. Маршруты, объявленные классами `@Endpoint`, поднимаются, когда аддон загружается, и снова удаляются, когда он выгружается, так что включение и отключение аддона аккуратно добавляет и удаляет его API.

::: tip Два контекста Spring и какие бины можно внедрять
Ваш аддон живёт в `pluginBeanContext` — *ваши* бины (ваши DAO, эндпоинты, слушатели). Собственные сервисы Pano живут в отдельном хост-контексте `applicationContext`: `DatabaseManager`, `AuthProvider`, `SetupManager`, `PluginDatabaseManager` и другие.

Хост-бины **нельзя** внедрять в ваши конструкторы — сканер видит только ваш пакет. Чтобы дотянуться до хост-сервиса, получайте его явно:

```kotlin
private val setupManager by lazy {
    applicationContext.getBean(SetupManager::class.java)
}
```

Вы дотягиваетесь до каждого хост-бина именно так — `applicationContext.getBean(...)` с типом бина.
:::

## Что происходит, когда загружается UI

UI никогда не поставляется в виде отдельных файлов. Когда вы запускаете релизную сборку, скомпилированные папки `plugin-ui/{client,server}` упаковываются в единственный ресурс jar, **`plugin-ui.zip`**. Дальше:

1. При загрузке Pano вычисляет **UI-хеш** этого zip и анонсирует `{ version, uiHash }` для вашего аддона через API информации о сайте, который тема и панель и так вызывают.
2. Тема (браузер) импортирует ваш **клиентский** бандл с запросом-инвалидатором кеша — `client.mjs?v=<uiHash>` — так что новая сборка делает недействительной старую кешированную копию. Node-процесс импортирует **серверный** бандл, `server/server.mjs`, для SSR.
3. После импорта Pano создаёт ваш класс, экспортированный по умолчанию, и вызывает его `onLoad()`. Это единственная точка входа, где вы регистрируете всё, что добавляет UI — хуки, страницы, ссылки навигации.

UI-хеш важен главным образом для **выпущенных** аддонов: новая версия поставляет новый zip, хеш меняется, и запрос-инвалидатор кеша подтягивает свежий бандл. Во время разработки механизм иной — с включённым [Режимом разработки](/ru/addon/getting-started/) Pano пересобирает ваш UI-zip с диска на каждый запрос (хеш становится маркером `dev-build`, а не хешем содержимого), а тема заново запрашивает этот бандл на каждый запрос, а не кеширует его. Именно это заставляет изменение `bun run dev` появляться по F5. Руководство [Разработка фронтенда](/ru/addon/frontend/) охватывает то, что вы делаете внутри `onLoad()`.

## Общая среда выполнения Svelte

Это самое важное, что нужно понять об UI, и оно объясняет правило, которое удивляет всех.

Ваш клиентский бандл **не содержит Svelte, `svelte-i18n` или `@panomc/sdk`.** Сборка на rollup намеренно оставляет эти импорты *внешними* — неразрешёнными — в поставляемом бандле. Во время работы хост (тема или панель) предоставляет карту импортов, которая разрешает каждый из них в стабильную заглушку `/runtime`, и каждая заглушка ре-экспортирует **собственный** живой экземпляр модуля хоста.

В результате Pano и все его аддоны разделяют **одну** среду выполнения Svelte и **один** экземпляр SDK: один и тот же планировщик эффектов, одни и те же хранилища, одни и те же контексты. Если бы ваш аддон упаковал собственную приватную копию Svelte или SDK, у этой копии было бы своё отдельное состояние, и гидратация — повторное подключение браузера к отрендеренному на сервере HTML — сломалась бы.

Отсюда жёсткое следствие: поскольку скомпилированный вывод Svelte гарантированно работает только с **той же самой** версией Svelte, на которой он будет выполняться, версия компилятора вашей сборки должна точно совпадать с версией хоста. Именно поэтому вы никогда не закрепляете Svelte сами — `@panomc/sdk` закрепляет правильную версию, а ваш `rollup.config.js` отказывается собираться при несовпадении.

::: warning Никогда не добавляйте `svelte` в ваш `package.json`
Версия Svelte приходит из закрепления `@panomc/sdk`, а не от вас. Если вы объявите `svelte` сами, переопределение может разойтись с версией, которую отдаёт хост Pano, и сборка защищается именно от этого: при несовпадении версий `rollup.config.js` печатает ошибку и **завершается с кодом 1** — сборка намеренно проваливается. Расхождение версий ломает гидратацию так, что это мучительно отлаживать, поэтому защита останавливает вас до того, как вы что-либо отправите. Удалите любую запись `svelte` и переустановите зависимости.
:::

Ещё одно, что это объясняет: *единственные* «голые» импорты, остающиеся внешними, — это Svelte, `svelte-i18n` и `@panomc/sdk` (и их подпути). Всё остальное, что вы импортируете — сторонний пакет вроде библиотеки графиков, — не имеет хост-заглушки и **должно быть упаковано** в вашу сборку. [Справочник API фронтенда](/ru/addon/api-reference/) перечисляет точный набор разрешённых «голых» спецификаторов.

## Где живут ваши данные во время работы

Во время работы ваш аддон обращается к двум очень разным местам хранения, и полезно держать их раздельно:

| Расположение | Что оно хранит | Кто в него пишет |
|---|---|---|
| `plugins/<pluginId>/` (каталог данных) | `config.conf`, загруженные файлы, всё, что ваш аддон сохраняет на диск | создаётся автоматически при первой загрузке; в него пишете вы (например, через `PluginConfigManager`) |
| Внутри jar (ресурсы) | `locales/*.json`, `logo.png`, `plugin-ui.zip` | вшито при сборке — **только для чтения** во время работы |

Каталог данных именуется по вашему `pluginId` (для Shoutbox — `plugins/pano-plugin-shoutbox/`) и переживает перезапуски — это место для состояния конкретной установки. В **выпущенном** jar эти ресурсы фиксируются в момент сборки jar. Однако во время **разработки** Pano отдаёт их с диска в реальном времени: с включённым Режимом разработки он читает `locales/*.json` прямо с диска и пересобирает ваш UI-zip с диска на каждый запрос — что как раз и заставляет работать цикл «правь-и-обновляй». Код Kotlin — единственное, что никогда не горячее: он требует пересборки и перезапуска Pano. (В разделе Начало работы есть полная таблица «горячее против пересборки».)

## Что дальше

Теперь, когда у вас есть ментальная модель, выберите сторону для сборки:

- **[Разработка бэкенда](/ru/addon/backend/)** — добавьте таблицу, откройте API, защитите его правом доступа, залогируйте действие.
- **[Разработка фронтенда](/ru/addon/frontend/)** — смонтируйте компонент на главной странице, добавьте страницу в панели, вызовите свой API.
- **[Конфигурация манифеста](/ru/addon/manifest/)** — ключи `gradle.properties`, которые становятся манифестом вашего jar.
