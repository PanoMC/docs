# Разработка фронтенда

**Что даёт вам эта страница:** единая точка входа, общая для фронтенда любого аддона — `src/main.js`, — плюс карта к двум тематическим страницам, которые строят сам UI.

У вашего аддона две половины: [бэкенд](/ru/addon/backend/) на Kotlin и фронтенд на **Svelte** (визуальная часть, которую пользователи видят и кликают). Когда вы собираете свой аддон, скомпилированный бэкенд на Kotlin и ваши собранные файлы Svelte архивируются вместе в один `.jar`-файл — весь этот файл *и есть* ваш аддон. Вы не делаете для этого ничего особенного; сборка справляется сама.

Pano запускает два отдельных сайта: **тему** (то, что посетители видят на `yoursite.com`) и **панель** (административную панель на `yoursite.com/panel`). Ваш один аддон добавляет UI к обоим. Один файл — `src/main.js` — является точкой входа для обоих.

Сквозной пример на этих страницах — **Shoutbox**: небольшой виджет недавних «выкриков» вверху главной страницы плюс панель, где администраторы ими управляют. Эта страница подключает `main.js`; две тематические страницы — [UI темы](/ru/addon/theme-ui/) и [UI панели](/ru/addon/panel-ui/) — строят собственно экраны (см. [Что дальше](#что-дальше)).

> Если ИИ-ассистент или старое руководство подсовывает вам API, которого нет на этих страницах или в [Справочнике API фронтенда](/ru/addon/api-reference/), — его не существует. Распространённые поддельные или удалённые вызовы перечислены [внизу этой страницы](#старые-и-выдуманные-ии-api-которых-не-существует).

::: tip Новичок в Svelte?
UI аддонов пишутся на Svelte, так же как и темы Pano. Если вы никогда его не использовали, интерактивное [руководство по Svelte](https://svelte.dev/tutorial) покрывает всё, что нужно UI аддона.
:::

::: tip Прежде чем начать
Вы должны были создать свой аддон из шаблона **pano-boilerplate-plugin** и завершить [Разработку бэкенда](/ru/addon/backend/). Файл `src/main.js` уже существует в boilerplate — вы будете его *редактировать*, а не создавать. Запустите цикл разработки командой `bun run dev` и держите его запущенным всё время; каждое изменение на этих страницах перезагружается на лету, так что вы сможете сразу его видеть. (Превращение готового аддона в релиз описано в [Сборке и публикации](/ru/addon/publishing/).)
:::

## Файлы вашего аддона

Вот раскладка, на которую ссылаются эти страницы. Папки `theme/` и `panel/` — просто аккуратное соглашение для того, чтобы держать компоненты для посетителей и компоненты для администраторов раздельно — Pano не навязывает имена, вы могли бы организовать файлы как угодно.

```text
src/
├─ main.js                    ← entry point; Pano loads this first
├─ theme/                     ← components shown to visitors (the public site)
│   └─ ShoutboxWidget.svelte
└─ panel/                     ← components shown to admins (the dashboard)
    ├─ ShoutboxSettings.svelte
    └─ ShoutboxPage.svelte
```

## Точка входа: `src/main.js`

Всё начинается в `main.js`. Он экспортирует по умолчанию класс, расширяющий `PanoPlugin`. Pano создаёт один экземпляр вашего класса, устанавливает на нём `this.pano` за вас (вы никогда не устанавливаете его сами), затем вызывает ваш метод `onLoad()`.

Он вызывает `onLoad()` **дважды** — один раз в панели и один раз в теме. Тема и панель — две отдельные работающие программы, и каждая загружает ваш аддон независимо, поэтому `onLoad()` действительно выполняется дважды, по разу в каждой. Так и должно быть, это не баг. Проверка `pano.isPanel` — это то, как вы даёте каждой стороне разный UI: она `true` в панели и `false` в теме.

Вот скелет, на котором строится весь аддон. Одна строка в нём — `export const _ = ...` — выглядит пугающе; пока скопируйте её как есть, это помощник для перевода, полностью объяснённый в [Переводе текста](#перевод-текста-в-ваших-компонентах) ниже, и вам не нужно его понимать, чтобы использовать скелет.

```js
// src/main.js
import { PanoPlugin, viewComponent } from '@panomc/sdk';
import ApiUtil from '@panomc/sdk/utils/api';
import { derived } from 'svelte/store';
import { _ as i18n } from '@panomc/sdk/utils/language';

export const pluginId = 'pano-plugin-shoutbox';

// A translate function scoped to your addon's keys — see the i18n section below.
export const _ = derived(i18n, ($t) => (key, options) => $t(`plugins.${pluginId}.${key}`, options));

export default class ShoutboxUiPlugin extends PanoPlugin {
  onLoad() {
    const { pano } = this;

    if (pano.isPanel) {
      // panel registrations go here
    } else {
      // theme registrations go here
    }
  }

  onUnload() {}
}
```

Несколько важных правил, которым следуют все встроенные аддоны:

- **Класс должен быть экспортом `default`.** Pano ищет ровно один.
- **`this.pano` — это весь ваш API.** Pano устанавливает его за вас до того, как выполнится `onLoad()` — внутри `onLoad()` вы просто читаете `pano.isPanel` и используете `pano.ui.*`.
- **`pluginId` должен в точности совпадать с ID вашего аддона** из [Разработки бэкенда](/ru/addon/backend/) / вашего манифеста плагина. Переводы и хук страницы деталей в панели привязаны к нему, поэтому несовпадение молча их ломает.
- **Оборачивайте каждый компонент в `viewComponent(() => import('./File.svelte'))`.** Это не необязательно. Каждый компонент, который вы передаёте вызову `register` — хуки, страницы, слоты представлений — должен быть обёрнут так. Простыми словами: это передаёт Pano *рецепт* загрузки вашего файла вместо самого файла, чтобы Pano мог загрузить его в правильный момент с собственной копией Svelte страницы. Детали вам не нужны — см. [Архитектуру](/ru/addon/architecture/), если любопытно.
- **Держите общее состояние в `main.js`.** Переменные, которые вы объявляете на верхнем уровне `main.js` (вроде `_`-хранилища выше), существуют ровно один раз и разделяются всеми вашими компонентами. Сборка гарантирует это только для `main.js`, поэтому не полагайтесь на другие файлы для общего состояния. (Механика — в [Архитектуре](/ru/addon/architecture/).)
- **`onUnload()` выполняется, когда ваш аддон отключается.** Пока оставьте его пустым; большинству аддонов он никогда не нужен.

::: warning Никогда не объявляйте `svelte` в `package.json`
Ваш бандл **не** поставляет Svelte, `svelte-i18n` или `@panomc/sdk` — их предоставляет хост, чтобы вся страница разделяла один экземпляр Svelte. Если ваша сборка начинает падать сразу после того, как вы (или библиотека, которую вы установили через `bun add`) добавили `svelte` в `package.json`, причина в этом: удалите `svelte` из `package.json`. Простыми словами, страница может запускать только одну копию Svelte, и хост уже её предоставляет; добавление собственной закрепляет вторую копию и ломает **гидрацию** (шаг, на котором серверно-отрисованный HTML привязывается, чтобы стать интерактивным в браузере). См. [Архитектуру](/ru/addon/architecture/) о том, почему.
:::

Теперь заполните две ветки. Что идёт в каждую — у каждой своя страница: **[UI темы](/ru/addon/theme-ui/)** для ветки `else` (тема), **[UI панели](/ru/addon/panel-ui/)** для ветки `if (pano.isPanel)`. Остальная часть этой страницы охватывает две вещи, которые используют обе ветки.

## Перевод текста в ваших компонентах

В отличие от разработки темы, **в компоненты вашего аддона автоматически не внедряется никакой помощник для перевода** — вы должны импортировать `_` из вашего собственного `main.js`. Этот `_` — та самая пугающая строка из скелета:

```js
export const _ = derived(i18n, ($t) => (key, options) => $t(`plugins.${pluginId}.${key}`, options));
```

Вот всё, что она делает: это `derived`-**хранилище** Svelte (значение, которое пересчитывается, когда меняется его источник), оборачивающее функцию перевода Pano, и оно автоматически добавляет к каждому переданному вами ключу префикс `plugins.<pluginId>.`. Так что в компоненте вы пишете `$_('widget.title')`, и оно ищет `plugins.pano-plugin-shoutbox.widget.title`. Вам не нужно разбираться в каррировании, чтобы это использовать, — просто импортируйте и читайте это с префиксом `$`:

```svelte
<script>
  import { _ } from '../main.js';
</script>

<h2>{$_('widget.title')}</h2>
```

`{$_('widget.title')}` разрешается в ключ `plugins.pano-plugin-shoutbox.widget.title` в ваших файлах локалей. См. [Локализацию](/ru/addon/localization/) о том, где живут эти ключи и как они разбиты по пространствам имён.

## Старые и выдуманные ИИ API, которых не существует

Если ИИ-инструмент, старое руководство или скаффолдинг предлагает что-либо из этого, игнорируйте — ничего из этого не существует. Используйте только то, что на этих страницах и в [Справочнике API фронтенда](/ru/addon/api-reference/).

- **`pano.ui.page.register({ name, view, scopes })`** — настоящий `page.register` принимает `{ path, component, permission, ... }` (см. [UI панели](/ru/addon/panel-ui/)). Формы с `name`/`view`/`scopes` не существует.
- **`import { Button, Card } from '@panomc/sdk/components/panel'`** — такой библиотеки компонентов в SDK нет.
- **`onContextUpdate`** — старый boilerplate определяет этот метод, но **ни один хост его не вызывает**. Если ваш созданный скаффолдингом `main.js` содержит `onContextUpdate`, удалите его.
- **`ApiUtil.get('/api/...')` с простой строкой** — каждый вызов `ApiUtil` принимает объект опций, например `ApiUtil.get({ path: '/api/...' })`.
- **`pano.utils.toast`** — такого нет; toast-ы приходят только из `@panomc/sdk/toasts`.

## Что дальше

`main.js` настроен. Теперь постройте UI:

- **Показать вещи на сайте → [UI темы](/ru/addon/theme-ui/)** — хуки темы, виджет главной страницы, серверно-отрисованные данные и вызов вашего API.
- **Добавить административные экраны → [UI панели](/ru/addon/panel-ui/)** — разделы настроек, полноценные страницы панели со ссылками навигации и toast-ы.
- **Полный поиск → [Справочник API фронтенда](/ru/addon/api-reference/)** — каждое имя хука, слот представления, событие жизненного цикла и экспорт `@panomc/sdk` в одном месте.
- **[Локализация](/ru/addon/localization/)** — где живут ваши строки `plugins.<pluginId>.<key>` и как панель позволяет администраторам их переопределять.
- **[Разработка бэкенда](/ru/addon/backend/)** — сторона Kotlin для эндпоинтов и прав доступа, которые вызывают эти страницы.
- **[Сборка и публикация](/ru/addon/publishing/)** — превратите готовый аддон в релизный jar. Релизная сборка **обязана** включать UI, поэтому никогда не используйте для неё `-Pnoui` (`-Pnoui` — это флаг Gradle, который пропускает сборку UI во время бэкенд-только итераций — см. Сборку и публикацию).
- **Аддоны для образца** — встроенные аддоны в [организации PanoMC на GitHub](https://github.com/PanoMC) — рабочий образец для каждого паттерна на этих страницах.
