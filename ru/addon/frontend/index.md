# Разработка фронтенда

Ваш аддон поставляет **UI на Svelte** в том же jar, что и его [бэкенд](/ru/addon/backend/). Этот UI выполняется в двух местах: в публичной **теме**, которую видят посетители, и в **панели**, которой пользуются администраторы. Один файл — `src/main.js` — является точкой входа для обоих.

Эта страница собирает фронтенд нашего сквозного примера, **Shoutbox**: небольшой виджет с недавними «выкриками» вверху главной страницы плюс панель, где администраторы управляют ими. К концу вы сможете:

- смонтировать компонент на главной странице и дать ему отрендеренные на сервере данные,
- добавить раздел настроек на страницу деталей вашего аддона в панели,
- зарегистрировать полноценную страницу панели с собственной ссылкой навигации,
- вызвать свой бэкенд-API и показать toast,
- перевести текст внутри ваших компонентов.

Каждая сигнатура на этой странице настоящая. (В старой версии этой страницы использовались вызовы вроде `pano.ui.page.register({ name, view, scopes })` и `import { Button, Card } from '@panomc/sdk/components/panel'` — **ничего из этого не существует.** Используйте только то, что показано здесь и в [Справочнике API фронтенда](/ru/addon/api-reference/).)

::: tip Не знакомы со Svelte?
UI аддонов пишутся на Svelte, так же как и темы Pano. Если вы никогда им не пользовались, интерактивный [учебник по Svelte](https://svelte.dev/tutorial) охватывает всё, что нужно UI аддона.
:::

## Точка входа: `src/main.js`

Всё начинается здесь. `main.js` экспортирует по умолчанию класс, расширяющий `PanoPlugin`. Pano создаёт один экземпляр, внедряет `this.pano` и вызывает ваш `onLoad()` — один раз в процессе панели и один раз в процессе темы. Вы различаете их через `pano.isPanel` и регистрируете свой UI в нужной ветке.

Вот скелет, на котором строится весь аддон:

```js
// src/main.js
import { PanoPlugin, viewComponent } from '@panomc/sdk';
import ApiUtil from '@panomc/sdk/utils/api';
import { derived } from 'svelte/store';
import { _ as i18n } from '@panomc/sdk/utils/language';

export const pluginId = 'pano-plugin-shoutbox';

// Функция перевода, привязанная к ключам вашего аддона — см. раздел i18n ниже.
export const _ = derived(i18n, ($t) => (key, options) => $t(`plugins.${pluginId}.${key}`, options));

export default class ShoutboxUiPlugin extends PanoPlugin {
  onLoad() {
    const { pano } = this;

    if (pano.isPanel) {
      // регистрации для панели идут сюда
    } else {
      // регистрации для темы идут сюда
    }
  }

  onUnload() {}
}
```

Несколько важных правил, которым следуют все эталонные плагины:

- **Класс должен быть экспортом `default`.** Pano ищет ровно один.
- **`this.pano` — это весь ваш API.** Он внедряется до того, как выполнится `onLoad()` — читайте `pano.isPanel` и используйте `pano.ui.*` изнутри `onLoad()`.
- **Оборачивайте каждый компонент в `viewComponent(() => import(...))`.** Это не опционально. Каждый компонент, который вы передаёте в вызов `register` — хуки, страницы, слоты представлений — должен быть обёрнут так. `viewComponent` учит хост загружать ваш компонент лениво и рендерить его с общей средой выполнения Svelte.
- **Держите общее состояние в `main.js`.** Всё на уровне модуля (как хранилище `_` выше) живёт здесь. Хост импортирует вашу **точку входа** с запросом-инвалидатором кеша `?v=<uiHash>`, тогда как ленивые чанки импортируют её без запроса — два URL, которые иначе вычислили бы точку входа дважды, — поэтому сборка проводит её через виртуальный фасад и принудительно помещает `main.js` в единственный общий чанк, который выполняется ровно один раз. (Остальные файлы — это обычные чанки без запроса, каждый и так уже загружен один раз; `main.js` — просто гарантированно единственный дом для общего состояния.)

::: warning Никогда не объявляйте `svelte` в `package.json`
Ваш бандл **не** поставляет Svelte, `svelte-i18n` или `@panomc/sdk` — хост предоставляет их, чтобы вся страница разделяла один экземпляр Svelte. Добавление `svelte` самостоятельно закрепляет вторую копию и ломает гидратацию; сборка настроена так, чтобы падать на этом. См. [Архитектуру](/ru/addon/architecture/), чтобы понять почему.
:::

Вы видели `onUnload()` выше, но не `onContextUpdate()`. Более старый boilerplate определяет `onContextUpdate` — **ни один хост его никогда не вызывает**, так что не помещайте туда логику.

## Тема: смонтируйте виджет на главной странице

Тема предоставляет именованные **хуки** — места, куда аддоны могут внедрять компонент. Чтобы показать Shoutbox на главной странице, зарегистрируйте компонент для хука `page:home:top`, в ветке `else` (тема):

```js
pano.ui.hook.register({
  name: 'page:home:top',
  component: viewComponent(() => import('./theme/ShoutboxWidget.svelte')),
});
```

Этого достаточно, чтобы отрисовать виджет. Но хорошему виджету нужны данные, и нужно, чтобы эти данные были в **первом ответе сервера** — а не подгружались позже в браузере, — чтобы посетители и поисковые системы видели выкрики сразу.

### Дайте виджету отрендеренные на сервере данные через `load()`

Компонент хука может экспортировать функцию `load(event)` из своего **module-скрипта**. Тема выполняет её, пока страница готовится (на сервере при SSR и на клиенте при навигации), и передаёт то, что вы вернули, компоненту как пропы. В Shoutbox `load()` вызывает наш публичный эндпоинт:

```svelte
<!-- src/theme/ShoutboxWidget.svelte -->
<script module>
  import ApiUtil from '@panomc/sdk/utils/api';

  export async function load(event) {
    const res = await ApiUtil.get({ path: '/api/shoutbox/list', request: event });
    return { shouts: res.shouts ?? [] };
  }
</script>

<script>
  export let shouts = [];
</script>

<div class="shoutbox">
  {#each shouts as shout}
    <p class="shout">{shout.message}</p>
  {/each}
</div>
```

Объект, который вы возвращаете, становится пропами компонента — здесь `shouts` приходит готовым к отрисовке. Это тот поток пропов, который хост называет **`hookProps`**.

::: warning `load()` выполняется на сервере *и* на клиенте
Один и тот же `load()` выполняется во время SSR и снова при навигации на стороне клиента. Держите его **идемпотентным** — без побочных эффектов и никогда не мутируйте объекты, которые вам передали. Всегда передавайте `request: event` в `ApiUtil` (следующий раздел), чтобы серверный вызов нёс сессию посетителя.
:::

Два дополнительных приёма, которые используют настоящие аддоны:

- **Условно скрыть виджет.** Если `load()` возвращает `{ hookOptions: { invisible: true } }`, хост ничего не рендерит для этого хука. Аддон Announcement использует это, чтобы исчезать, когда показывать нечего.
- **Пропустить `load()` и получить данные в другом месте.** Зарегистрируйтесь с `skipLoad: true` и получайте данные из `pano.ui.app.onLoad(async (data, event) => { ... })` вместо этого. Это альтернативный путь данных — аддоны FAQ и Pages используют его, когда один запрос питает несколько регистраций.

## Панель: раздел настроек на странице деталей вашего аддона

Когда администратор открывает ваш аддон в **Панель → Аддоны**, у его страницы деталей есть хук с именем `panel:plugin-detail:content:<pluginId>`. Регистрация компонента там — самый дешёвый способ дать вашему аддону экран настроек — большинство встроенных аддонов делают именно это. Поместите это в ветку `if (pano.isPanel)`:

```js
pano.ui.hook.register({
  name: `panel:plugin-detail:content:${pluginId}`,
  component: viewComponent(() => import('./panel/ShoutboxSettings.svelte')),
});

pano.ui.addon.onLoad(async (data, event) => {
  if (data.addon.id !== pluginId) return;
  const res = await ApiUtil.get({ path: '/api/panel/shoutbox/config', request: event });
  if (!res.error) data.addon.config = res;
});
```

`pano.ui.addon.onLoad` срабатывает, когда загружается страница деталей любого аддона, поэтому первая строка проверяет, что это **ваш**, прежде чем получать данные. Конфигурация, которую вы прикрепляете к `data.addon.config`, затем доступна `ShoutboxSettings.svelte` через проп `addon`:

```svelte
<!-- src/panel/ShoutboxSettings.svelte -->
<script>
  export let addon;
  let config = addon?.config ?? { enabled: true, maxShouts: 5 };
</script>
```

## Панель: полноценная страница с собственной ссылкой навигации

Раздел настроек живёт внутри страницы деталей аддона. Когда вашему аддону нужна собственная страница — экран управления по адресу `/shoutbox` — зарегистрируйте её как **страницу** и добавьте ссылку в боковую панель панели:

```js
pano.ui.page.register({
  path: '/shoutbox',
  component: viewComponent(() => import('./panel/ShoutboxPage.svelte')),
  permission: 'pano.plugin.pano-plugin-shoutbox.manage.shoutbox',
});

pano.ui.nav.site.editNavLinks(async (links) => {
  if (!links.some((l) => l.href === '/shoutbox')) {
    const i = links.findIndex((l) => l.href === '/posts');
    const link = {
      href: '/shoutbox',
      icon: 'fas fa-bullhorn',
      text: `plugins.${pluginId}.nav.shoutbox`,
      startsWith: true,
      permission: 'pano.plugin.pano-plugin-shoutbox.manage.shoutbox',
    };
    i >= 0 ? links.splice(i + 1, 0, link) : links.push(link);
  }
  return links;
});
```

Обратите внимание на два соглашения, которым следует каждый аддон во флоте:

- **Вставляйте относительно якоря.** Найдите существующую ссылку (здесь `/posts`) и вставьте свою ссылку рядом с ней, откатываясь к `push`, если якоря нет, — так ваша ссылка окажется в разумном месте, а не всегда в конце.
- **Защищайтесь от дубликатов.** `editNavLinks` может выполниться более одного раза, поэтому проверяйте `links.some((l) => l.href === '/shoutbox')`, прежде чем добавлять. Всегда **возвращайте** массив.

::: tip Держите строку права доступа синхронизированной
Строка `permission` выше — это набранная вручную копия узла, который выводит ваш Kotlin-класс `ManageShoutboxPermission` (`pano.plugin.pano-plugin-shoutbox.manage.shoutbox`). Общей константы нет — если вы переименуете класс Kotlin, выводимый узел изменится, и этот UI-запрет молча перестанет совпадать. Меняйте оба вместе. См. правило вывода права доступа в [Разработке бэкенда](/ru/addon/backend/).
:::

Если `permission` не выполнено, страница возвращает 404, а ссылка навигации скрывается. Для страниц есть ещё две опции: `systemLayout` переиспользует встроенный макет панели (аддон Comments использует `systemLayout: 'PostsLayout'`, так что его страница располагается под разделом Posts), а `resetLayout` убирает окружающее оформление. Полный список имён макетов — в [Справочнике API фронтенда](/ru/addon/api-reference/).

## Вызов вашего API и показ toast'ов

Все сетевые вызовы идут через `ApiUtil`. Импортируйте экспорт по умолчанию и используйте методы-глаголы, каждый из которых принимает единственный объект опций:

```js
import ApiUtil from '@panomc/sdk/utils/api';

// Внутри load() — передавайте request, чтобы серверный вызов имел сессию:
const res = await ApiUtil.get({ path: '/api/shoutbox/list', request: event });

// В обработчике события браузера — body это ваша JSON-нагрузка:
await ApiUtil.post({ path: '/api/panel/shoutbox', body: { message } });
await ApiUtil.delete({ path: `/api/panel/shoutbox/${id}` });
await ApiUtil.put({ path: '/api/panel/shoutbox/config', body: config });
```

Правило: **внутри `load()` всегда передавайте `request: event`**, чтобы запрос выполнялся с сессией посетителя во время SSR. В обработчике клика, выполняющемся в браузере, вы можете его опустить.

Чтобы подтвердить действие, покажите toast:

```svelte
<script>
  import { showToast } from '@panomc/sdk/toasts';
  import { _ } from '../main.js';

  async function save(config) {
    const res = await ApiUtil.put({ path: '/api/panel/shoutbox/config', body: config });
    showToast(res.error ? $_('toasts.save-error') : $_('toasts.save-success'));
  }
</script>
```

::: warning Этих старых сигнатур не существует
`ApiUtil.get('/api/...')` с простой строкой — это **неправильно** — каждый вызов принимает объект опций. Также **нет `pano.utils.toast`**; toast'ы приходят только из `@panomc/sdk/toasts`.
:::

## Перевод текста в ваших компонентах

Ничего не внедряется в ваши компоненты автоматически. Помощник перевода — это хранилище `_`, которое вы экспортировали из `main.js` в начале этой страницы, — `derived`-хранилище, добавляющее к каждому ключу префикс `plugins.<pluginId>.`. Импортируйте его и читайте с префиксом `$`:

```svelte
<script>
  import { _ } from '../main.js';
</script>

<h2>{$_('widget.title')}</h2>
```

`{$_('widget.title')}` разрешается в ключ `plugins.pano-plugin-shoutbox.widget.title` в ваших файлах локалей. См. [Локализацию](/ru/addon/localization/) о том, где живут эти ключи и как они разделяются по пространствам имён.

## Динамические страницы и очистка

Иногда страницы, которые вы регистрируете, неизвестны на этапе сборки — они приходят из вашего бэкенда (пользовательские URL, редиректы и тому подобное). Регистрируйте их изнутри `pano.ui.app.onLoad`, после получения списка. Аддон Pages делает именно это.

Здесь есть острый угол. Процесс SSR темы **долгоживущий**: `pano.ui.app.onLoad` выполняется на каждый запрос, но маршруты и ссылки, которые вы регистрируете, сохраняются в этом процессе между запросами. Если страница удалена в панели, запись, которую вы зарегистрировали ранее, задерживается — SSR продолжает отдавать призрачный маршрут до перезапуска процесса, хотя браузер о нём больше не знает.

Исправление — отслеживать то, что вы зарегистрировали, и удалять записи, которых больше нет, через `pano.ui.page.unregister(path)`:

```js
const registeredPaths = new Set();
const customPageComponent = viewComponent(() => import('./theme/CustomPage.svelte'));

pano.ui.app.onLoad(async (data, event) => {
  const res = await ApiUtil.get({ path: '/api/pages', request: event });
  const incoming = new Set(res.pages.map((p) => p.url));

  // Удаляем зарегистрированные ранее маршруты, которых больше нет.
  for (const path of registeredPaths) {
    if (!incoming.has(path)) {
      pano.ui.page.unregister(path);
      registeredPaths.delete(path);
    }
  }

  for (const page of res.pages) {
    pano.ui.page.register({ path: page.url, component: customPageComponent });
    registeredPaths.add(page.url);
  }
});
```

::: warning Призрачные маршруты в SSR
Динамические страницы, зарегистрированные из данных, выживают в Node-процессе. Если вы никогда не снимаете регистрацию удалённых элементов, удалённые страницы продолжают отдаваться во время SSR до перезапуска Pano. Аддон `pano-plugin-link-redirects` — полный эталон этого шаблона очистки, включая удаление устаревших ссылок навигации, которые вы добавили.
:::
