# Начало работы

**Аддон** Pano добавляет возможности сайту Pano: новые страницы в панели, новые разделы в теме, новые API на бэкенде — и всё это в одном устанавливаемом файле. Вы собираете его из готового шаблона и помещаете в работающий Pano.

::: tip Аддон и плагин — это одно и то же
Аддоны — это *плагины* Pano: API уровня кода, имена папок и классы — все используют слово `plugin` (например, `PanoPlugin`, `pluginId`). На этой странице (и в остальной документации) в тексте говорится **аддон**, но в коде вы постоянно будете видеть `plugin`. Так и задумано; ничего не переименовано.
:::

Аддон — это единственный JAR-файл, содержащий две части, которые работают вместе:

- **бэкенд на Kotlin** (плагин PF4J), который выполняется внутри сервера Pano — он добавляет таблицы базы данных, JSON-API, права доступа и многое другое;
- **UI-бандл на Svelte**, который выполняется в браузере, **как** в панели администратора, **так и** в активной теме.

Использовать обе части необязательно — аддон только с UI или только с бэкендом вполне допустим, — но шаблон поставляется с уже связанными между собой обеими частями. Чтобы точно понять, где выполняется каждый кусок, прочитайте далее [Архитектуру](/ru/addon/architecture/). Если вы просто хотите *устанавливать* аддоны, а не создавать их, см. пользовательскую страницу [Аддоны](/ru/platform/addons/).

Эта страница проведёт вас от нуля до собственного аддона, отображающегося в **Панель → Аддоны**, с рабочим циклом «правь-и-обновляй». Наш сквозной пример в этих руководствах — небольшой аддон под названием **Shoutbox**: посетители видят последние «выкрики» (shouts) на главной странице, а администраторы управляют ими из панели, — поэтому мы с самого начала назовём наш проект `pano-plugin-shoutbox`.

## Что вам нужно

| Требование | Примечания |
|---|---|
| **JDK 11 или новее** | Аддон компилируется под **набор инструментов Java 11**, поэтому он работает на любом сервере, который поддерживает Pano (самому Pano нужна лишь Java 11+). Gradle может работать на JDK 11 или новее; если ваш JDK новее 11, шаблон автоматически скачает набор инструментов Java 11 за вас. |
| **Bun** | Менеджер пакетов и сборщик UI. Установите его с [bun.sh](https://bun.sh). Gradle автоматически скачивает Bun 1.2.0 для релизных сборок, но вам нужно, чтобы он был установлен локально, чтобы можно было запускать `bun run dev`. |
| **Запущенный экземпляр Pano** | Самостоятельно размещённый, с уже завершённым мастером установки. Ваш аддон живёт внутри этого экземпляра, пока вы работаете. См. [Установку](/ru/platform/installation/), если вы её ещё не выполнили. |
| **Режим разработки (Development Mode)** | Переключатель в **Панель → Настройки платформы** (ключ конфигурации `development-mode`). Он отключает кеширование UI аддонов в теме (так что ваш бандл заново запрашивается на каждый запрос) и заставляет файлы локалей загружаться с диска в реальном времени. |
| **Pano, запущенный в среде разработки** | Живой цикл обновления UI работает, только когда *процесс* Pano запущен в среде DEVELOPMENT — dev-сборка Pano, либо платформа, запущенная с переменной окружения `EnvironmentType=DEVELOPMENT`. На обычном релизном экземпляре бэкенд всегда отдаёт UI-бандл, вшитый в ваш jar, поэтому вывод `bun run dev` не появится при обновлении, даже если Режим разработки включён. |
| **`init-ui = true`** | Настройка конфигурации, которая говорит Pano запускать свои UI-движки (setup, панель, тема) при старте — без них нет ни панели, ни темы, куда можно было бы загрузить UI вашего аддона. Она **по умолчанию `true`**, поэтому обычно её трогать не нужно; просто убедитесь, что её не отключили. |

::: tip Два переключателя, а не один
Для живого цикла обновления UI нужны **оба**: включённый Режим разработки *и* запущенный в среде разработки Pano. Режим разработки (переключатель конфигурации) отключает кеширование UI аддонов в теме и включает загрузку локалей с диска в реальном времени. Среда **разработки** (dev-сборка Pano или `EnvironmentType=DEVELOPMENT` в процессе) — это то, что заставляет бэкенд собирать ваш UI-zip заново с диска на каждый запрос вместо того, чтобы отдавать копию, вшитую в jar. На обычном релизном экземпляре одного лишь включения Режима разработки недостаточно — бэкенд всё равно отдаёт бандл из jar. Включите оба, прежде чем начинать.
:::

## Создание аддона из шаблона

Pano поставляет готовый шаблон, [`pano-boilerplate-plugin`](https://github.com/PanoMC/pano-boilerplate-plugin), в котором бэкенд и UI уже связаны между собой. Вы клонируете его, а затем переименовываете всё под свой собственный аддон.

**Клонируйте его в папку `plugins/` вашего экземпляра Pano.** Это важно: аддон перезагружает свой UI на лету, только когда он находится внутри папки `plugins/` работающего экземпляра.

```bash
cd <your-pano-instance>/plugins
git clone https://github.com/PanoMC/pano-boilerplate-plugin.git pano-plugin-shoutbox
cd pano-plugin-shoutbox
```

Теперь переименуйте шаблон под свой аддон. Шаблон называет себя `pano-boilerplate-plugin` в нескольких местах — измените каждое из них. Вот полный чек-лист с точными значениями для нашего примера Shoutbox:

| Расположение | Что изменить |
|---|---|
| `gradle.properties` | `pluginId` → `pano-plugin-shoutbox`; `pluginName` → `Shoutbox`; `pluginDescription`, `pluginDeveloper`, `pluginLicense`, `pluginSourceUrl`, `organization` → ваши собственные значения; `pluginClass` → `com.panomc.plugins.shoutbox.ShoutboxPlugin` |
| Пакет Kotlin | Переименуйте папку `src/main/kotlin/com/panomc/plugins/boilerplate` → `.../shoutbox` и обновите строку `package` внутри файла |
| Главный класс Kotlin | `BoilerplatePlugin` → `ShoutboxPlugin` (должен совпадать с `pluginClass` выше) |
| `src/main.js` | Константу `pluginId` `'pano-boilerplate-plugin'` → `'pano-plugin-shoutbox'`, а имя класса по умолчанию `PanoExamplePlugin` → ваше собственное (например, `ShoutboxUiPlugin`) |
| `package.json` | `"name"` → `pano-plugin-shoutbox` |
| `settings.gradle.kts` | Добавьте `rootProject.name = "pano-plugin-shoutbox"` — шаблон поставляет этот файл **пустым**, поэтому без этой строки имя проекта Gradle откатывается к имени папки |
| `src/main/resources/locales/en-US.json` | Ваши текстовые строки (в шаблоне один ключ, `hello-world`) |
| `src/main/resources/logo.png` | Замените на свой логотип |

::: tip `pluginId` используется повсюду
`pluginId`, который вы задаёте в `gradle.properties`, — это не просто метка: он становится именем каталога данных, сегментом URL вашего UI, префиксом узлов прав доступа и идентификатором ресурса на маркетплейсе. Выберите его один раз и держите неизменным. Полные сведения о манифесте — на странице [Конфигурация манифеста](/ru/addon/manifest/).
:::

## Первая сборка и проверка

Установите зависимости UI и соберите весь аддон один раз:

```bash
bun install
./gradlew build
```

Это создаст JAR в каталоге `build/libs/` (с именем `pano-plugin-shoutbox-local-build.jar`). Pano обнаруживает только jar-файлы, лежащие **непосредственно** в папке `plugins/` экземпляра — он *не* сканирует вложенный каталог `build/libs/`, в котором находится ваш клон, — поэтому скопируйте только что собранный jar на уровень выше:

```bash
cp build/libs/pano-plugin-shoutbox-local-build.jar ..   # в папку plugins/ экземпляра
```

Затем перезапустите Pano и откройте **Панель → Аддоны** — ваш аддон должен быть там в списке. Это и есть проверка «загрузился ли он?». (Новые jar-файлы подхватываются только при запуске; действия панели над аддоном — это включить / отключить / удалить / загрузить, а не повторное сканирование.) Если он появился, значит бэкенд-часть загрузилась корректно и вы готовы к итерациям.

::: tip Клон и загружаемый jar — это две разные вещи
Ваш клон остаётся по пути `plugins/pano-plugin-shoutbox/` — именно этот путь (совпадающий с вашим `pluginId`) Pano читает для живого UI и файлов локалей во время разработки. **Загружаемый** бэкенд-jar — это копия, которую вы кладёте непосредственно в `plugins/`. Оба прекрасно сосуществуют.
:::

## Цикл разработки

Это самая важная часть страницы. Во время разработки вы почти никогда не захотите запускать полную `./gradlew build` — она каждый раз пересобирает UI, что медленно. Вместо этого используйте **две команды**, по одной на каждую половину аддона.

Для работы над **бэкендом (Kotlin)** соберите быстрый бэкенд-только JAR, пропускающий UI:

```bash
./gradlew build -Pnoui   # быстрый бэкенд-только jar, пропускает сборку UI
```

Для работы над **UI (Svelte)** запустите наблюдатель и оставьте его работать:

```bash
bun run dev              # rollup watch → src/main/resources/plugin-ui/{client,server}
```

`bun run dev` перекомпилирует ваш UI в папку `plugin-ui` аддона каждый раз, когда вы сохраняете. Пока Режим разработки включён, Pano запущен в среде разработки, а аддон находится в папке `plugins/` экземпляра, обновление браузера немедленно подхватывает новую сборку — без пересборки JAR.

Впрочем, не каждое изменение так быстро. Вот что именно требуется для каждого вида изменений:

| Что вы изменили | Что нужно, чтобы это увидеть |
|---|---|
| UI на Svelte (`src/main.js`, `src/panel/**`, `src/theme/**`) | Запущенный `bun run dev` + обновление браузера (F5) |
| `locales/*.json` | При включённом Режиме разработки — обновление браузера (F5); локали читаются с диска в реальном времени из вашего дерева исходников |
| Код Kotlin | `./gradlew build -Pnoui`, скопировать новый jar в `plugins/`, затем **перезапустить Pano** |
| `gradle.properties`, исходный `config.conf` | Полная `./gradlew build`, скопировать jar в `plugins/`, затем перезапустить Pano |

::: warning Изменения в Kotlin требуют пересборки *и* перезапуска
Код Kotlin **не** горячий. После редактирования файла `.kt` пересоберите (`./gradlew build -Pnoui` достаточно для кода только бэкенда), скопируйте новый jar в папку `plugins/` вашего экземпляра и **перезапустите Pano**. Отключение и повторное включение аддона в панели *не* подхватывает пересобранный байт-код — PF4J держит уже загруженный загрузчик классов, поэтому только перезапуск загружает новый jar. Изменения в `gradle.properties` или исходном `config.conf` требуют полной `./gradlew build` тем же образом.
:::

::: warning Никогда не проксируйте `/plugins` в конфиге Vite/dev
Если вы также запускаете dev-сервер темы или панели, **не** добавляйте правило прокси для `/plugins` в его конфиг. UI-сервер уже отдаёт этот путь; проксирование его обратно создаёт цикл запросов, который подвесит ваш dev-сервер.
:::

## Ваше первое изменение за 2 минуты

Давайте сделаем по одному изменению каждого вида, чтобы разница «горячее против пересборки» действительно уложилась.

**Изменение UI (горячее — достаточно F5).** Откройте `src/main.js` и добавьте строку лога внутрь `onLoad()`:

```js
onLoad() {
  const pano = this.pano;
  console.log('Shoutbox UI loaded! isPanel =', pano.isPanel);
  // ...
}
```

Убедитесь, что `bun run dev` запущен, сохраните файл, затем обновите ваш сайт Pano и откройте консоль браузера. Ваше сообщение там — пересборка не нужна.

**Изменение локали (тоже горячее в Режиме разработки).** Откройте `src/main/resources/locales/en-US.json` и измените единственную строку шаблона:

```json
{
  "hello-world": "Hello from Shoutbox!"
}
```

Сохраните и нажмите F5. Поскольку Режим разработки включён, а ваш клон находится в `plugins/<pluginId>/`, Pano читает файлы локалей **с диска в реальном времени** из вашего дерева исходников — так что новый текст появляется без пересборки, точно как UI.

**Изменение Kotlin (требует пересборки + перезапуска).** Это то самое, что *не* горячее. Измените любой файл `.kt`, выполните `./gradlew build -Pnoui`, скопируйте jar в `plugins/` и перезапустите Pano — отключение → включение в панели продолжит выполнять старый байт-код.

Вот и весь цикл в двух словах: **UI и локали живые в Режиме разработки; изменения в Kotlin и манифесте требуют пересборки и перезапуска Pano.** Держите это разделение в голове, и остальные руководства будут ощущаться быстрыми.

## Что дальше

Теперь у вас есть свой загруженный аддон, и вы знаете, какие изменения горячие, а какие нет. Вот рекомендуемый путь:

- **[Архитектура](/ru/addon/architecture/)** — ментальная модель: что происходит, когда Pano загружает ваш JAR, и где оказывается каждый файл во время работы. Прочитайте это до написания настоящего кода.
- **[Разработка бэкенда](/ru/addon/backend/)** — соберите бэкенд Shoutbox: таблицу базы данных, JSON-API, право доступа и журнал активности.
- **[Разработка фронтенда](/ru/addon/frontend/)** — соберите UI Shoutbox: смонтируйте виджет на главной странице, добавьте раздел настроек в панели и полноценную страницу в панели.
- **[Сборка и публикация](/ru/addon/publishing/)** — превратите аддон в релиз и отправьте его на маркетплейс.
