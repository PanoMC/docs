# Локализация (i18n)

Локализация (часто сокращается до **i18n**) означает показ текста вашего аддона на разных языках. Каждая строка, которую ваш аддон выводит на экран — кнопки панели, заголовки прав доступа, строки на странице Активности — живёт в JSON-файлах внутри jar, по одному файлу на язык.

Ваши файлы перевода живут в `src/main/resources/locales/`, по одному на язык:

```
src/main/resources/locales/
├─ en-US.json
├─ tr.json
└─ ru.json
```

- **Формат файла:** каждый файл — это обычный, валидный `.json`.
- **Имя файла — это код локали:** `en-US.json` — это код `en-US`, `tr.json` — это `tr`, и так далее.

::: warning en-US — это резервный вариант
Если ключ отсутствует для языка посетителя, Pano откатывается к `en-US.json`. Держите этот файл присутствующим и полным — это страховочная сетка для всех остальных языков.
:::

## Пространство имён — где живут ваши ключи

Платформа обслуживает каждый написанный вами ключ под `plugins.<pluginId>.<key>`. Для нашего примера-аддона **Shoutbox** (`pluginId` = `pano-plugin-shoutbox`) ключ с именем `widget.title` становится `plugins.pano-plugin-shoutbox.widget.title` во время поиска.

Вы никогда не пишете этот префикс сами. Внутри ваших JSON-файлов ключи остаются **без префикса** (`widget.title`); Pano добавляет часть `plugins.<pluginId>.` автоматически. Именно это удерживает ваши ключи от столкновений с ключами основной платформы или другого аддона.

## Редактирование локалей: горячо при разработке, вшито в релизе

Как изменение локали доходит до экрана, зависит от **Режима разработки**:

- **С включённым Режимом разработки** (и вашим аддоном, клонированным в `plugins/<pluginId>/` экземпляра, как настраивает [Начало работы](/ru/addon/getting-started/)) Pano читает ваши `locales/*.json` **с диска в реальном времени** на каждый запрос. Отредактируйте файл, обновите (F5) — и новый текст появляется без пересборки, точно как UI на Svelte.
- **С выключенным Режимом разработки или в выпущенном jar** файлы локалей — это вшитые **ресурсы**. Обновление не подхватит изменение; вам нужно пересобрать jar:

```bash
./gradlew build -Pnoui
```

затем скопировать jar в `plugins/` и **перезапустить Pano** (отключение и повторное включение аддона не перезагружает содержимое jar). См. [Начало работы](/ru/addon/getting-started/) для полной таблицы «горячее против пересборки».

::: tip Зачем два режима?
В production локали поставляются внутри jar, так что аддон остаётся одним самодостаточным файлом. Режим разработки меняет это на чтение с диска в реальном времени, чтобы вы могли итерировать переводы без пересборки — по той же причине он заново отдаёт ваш UI на Svelte на каждый запрос.
:::

## Пользовательская настройка и переопределения

Одна из приятных особенностей Pano: администраторы могут редактировать переводы вашего аддона прямо из панели.

- **Переопределения:** администратор может изменить любой определённый вами ключ — новая формулировка без прикосновения к вашему jar.
- **Новые языки:** администратор может добавить язык, который ваш аддон не поставляет, переведя ваши ключи в панели.

Эти правки хранятся Pano отдельно и переживают обновления аддона, так что формулировки владельца сайта никогда не теряются при выпуске новой версии.

## Определение переводов

### Пользовательские ключи

Это обычные пары ключ–значение, которые использует ваш UI. Вкладывайте их как вам угодно:

```json
{
  "widget": {
    "title": "Latest shouts",
    "empty": "No shouts yet — be the first!"
  },
  "welcome-message": "Welcome back, {username}!"
}
```

Плейсхолдеры используют **одинарные** фигурные скобки (`{username}`), потому что UI отрисовывает их с помощью svelte-i18n. (Есть одно исключение на бэкенде — см. заметку [Бэкенд](#бэкенд-kotlin) ниже.)

### Права доступа

Если ваш аддон определяет право доступа, дайте ему человекочитаемый заголовок и описание, чтобы администраторы могли понять его на странице **Права доступа** панели.

**Определение:**

```kotlin
@PermissionDefinition
class ManageShoutboxPermission : PanelPermission("fa-bullhorn")
```

**Перевод:**

```json
{
  "permissions": {
    "MANAGE_SHOUTBOX": {
      "title": "Manage Shoutbox",
      "description": "Allows managing shouts shown on the home page."
    }
  }
}
```

Ключ детерминирован: возьмите имя класса права доступа, отбросьте суффикс `Permission` и преобразуйте в `UPPER_SNAKE_CASE`.

- `ManageShoutboxPermission` → `MANAGE_SHOUTBOX`

### Журналы активности

Если ваш аддон записывает действия администратора, определите шаблон для каждого под `activity-logs`. Эти строки появляются на странице **Активность** панели.

**Определение:**

```kotlin
class CreatedShoutLog(
    userId: Long,
    username: String,
    pluginId: String,
    message: String,
) : PluginActivityLog(
    userId = userId,
    pluginId = pluginId,
    details = JsonObject().put("target", message).put("username", username),
)
```

**Перевод:**

```json
{
  "activity-logs": {
    "CREATED_SHOUT": "<b>{username}</b> posted a shout: {target}."
  }
}
```

Ключ следует тому же правилу, что и права доступа, но отбрасывает суффикс `Log` вместо этого:

- `CreatedShoutLog` → `CREATED_SHOUT`

Плейсхолдеры `{username}` и `{target}` заполняются из JSON `details` журнала — Kotlin-класс поставляет *данные*, а эта запись локали поставляет *формулировку*.

::: info Оба живут в корне вашего файла
`permissions` и `activity-logs` должны находиться в **корне** вашего JSON-файла, рядом с вашими пользовательскими ключами — а не быть вложенными в другой объект. Pano читает эти два раздела напрямую, чтобы связать их со страницами Прав доступа и Активности ядра.
:::

## Использование переводов

### Фронтенд (Svelte)

Ничего не внедряется в ваши компоненты автоматически. Ваш `src/main.js` экспортирует небольшое `derived`-хранилище, которое оборачивает svelte-i18n и добавляет за вас пространство имён вашего аддона:

```js
// src/main.js
import { derived } from 'svelte/store';
import { _ as i18n } from '@panomc/sdk/utils/language';

export const pluginId = 'pano-plugin-shoutbox';
export const _ = derived(i18n, ($t) => (key, options) => $t(`plugins.${pluginId}.${key}`, options));
```

Затем любой компонент импортирует этот `_` и использует его с **коротким, без префикса** ключом:

```svelte
<script>
  import { _ } from '../main.js';
</script>

<h2>{$_('widget.title')}</h2>
```

`$_('widget.title')` ищет за вас `plugins.pano-plugin-shoutbox.widget.title`. См. [Разработку фронтенда](/ru/addon/frontend/) о том, где это хранилище настраивается и используется.

::: tip Доступ к ключам вне вашего пространства имён
Если вам нужен ключ, который *не* находится под пространством имён вашего аддона, импортируйте собственное хранилище svelte-i18n — `import { _ } from '@panomc/sdk/utils/language'` — и передайте **полный путь** сами (например, `$_('plugins.pano-plugin-shoutbox.widget.title')`). `_` в вашем `main.js` — это просто то же самое хранилище с добавленным префиксом.
:::

### Бэкенд (Kotlin)

Две возможности бэкенда читают ваши файлы локалей:

- **Журналы активности.** Когда ваш аддон записывает действие администратора, вы вставляете подкласс `PluginActivityLog` (см. [Разработку бэкенда](/ru/addon/backend/)). Страница Активности отрисовывает его, ища `activity-logs.<KEY>` и заполняя плейсхолдеры `{...}` из полезной нагрузки `details` журнала. Локализованный текст живёт *здесь*, в файле локали — никогда в Kotlin.
- **Почта.** Если ваш аддон отправляет письма, тема и тело приходят из Handlebars-шаблонов.

::: warning Одинарные против двойных фигурных скобок
Всё, что отрисовывает UI — пользовательские ключи, заголовки прав доступа, строки журнала активности — проходит через svelte-i18n и интерполирует с **одинарными** фигурными скобками: `{username}`. **Единственное** место, где вы пишете **двойные** фигурные скобки `{{variable}}`, — это внутри бэкенд-шаблонов писем, которые Pano отрисовывает через Handlebars. Использование неверного стиля оставляет сырые `{...}` видимыми на экране.
:::
