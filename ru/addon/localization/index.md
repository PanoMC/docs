# Локализация (i18n)

**Что даёт вам эта страница:** к концу у вас будет файл локали, одна переведённая строка, показанная на экране, и вы будете знать, как добавлять больше языков, переводить названия прав доступа и переводить строки журнала активности.

Локализация/интернационализация (принятое в экосистеме сокращение — **i18n**, от *internationalization*) означает показ текста вашего аддона на разных языках. Каждый кусочек текста, который показывает ваш аддон, живёт в JSON-файлах, упакованных внутрь jar, — по одному файлу на язык. (`.jar` — это единственный zip-подобный файл, в который упакован аддон на Kotlin/Java, как tarball npm-пакета; вы никогда не открываете его вручную.)

::: tip Мне вообще надо это делать?
Да, хотя бы немного. Вы *можете* захардкодить английские строки прямо в свои Svelte-компоненты — но двум видам текста больше негде жить: **названия прав доступа** (показываемые на странице Права доступа панели) и **строки журнала активности** (показываемые на странице Активность панели) могут приходить *только* из файлов локалей. Так что даже аддону только на английском нужен `en-US.json`. Оба этих вида вы встретите ниже.
:::

## Настройте файлы локалей

Внутри проекта вашего аддона (того, что создало [Начало работы](/ru/addon/getting-started/)) создайте папку `src/main/resources/locales/`, если её ещё нет, и добавьте файл `en-US.json`, содержащий просто `{}`. (`resources` — это имя Gradle для не-кодовых файлов — JSON, изображений и подобного, — которые упаковываются в jar. Несмотря на имя, это не только для изображений.)

В итоге у вас получается по одному файлу на язык, где имя файла — это код этого языка:

```
src/main/resources/locales/
├─ en-US.json
├─ tr.json
└─ ru.json
```

Обязателен только `en-US.json`. Добавляйте другие языки (`tr.json`, `ru.json`, …), когда будете готовы переводить.

::: tip Контрольная точка
Теперь у вас на диске должен быть `src/main/resources/locales/en-US.json`, содержащий `{}`.
:::

- **Формат файла:** каждый файл — это обычный, валидный `.json` — без комментариев и без завершающих запятых (любое из двух делает весь файл невалидным), и сохраняйте его как **UTF-8**, чтобы турецкие или русские символы не превращались в искажённый `Ã¶`-подобный текст.
- **Имя файла — это код локали:** `en-US.json` — это код `en-US`, `tr.json` — это `tr` и так далее. Pano принимает короткие коды вроде `en-US` или `tr` — те же коды, что вы задаёте как язык сайта (см. [Конфигурацию](/ru/platform/configuration/)).

**Какой язык видит посетитель.** У сайта есть язык по умолчанию (настройка `locale`), и когда администратор это разрешает, каждый посетитель может выбрать свой из тех языков, что вы поставляете. Чтобы протестировать свой `tr.json`, переключите свой собственный язык на турецкий в **Панель → Настройки → Платформа → Предпочтения** (или через селектор языка сайта) и обновите.

::: warning en-US — это запасной вариант
Если ключа нет для языка посетителя, Pano откатывается к `en-US.json`. Если ключа нет *и там тоже*, посетитель видит на экране сырой путь ключа — что-то вроде `plugins.pano-plugin-shoutbox.widget.title` вместо настоящих слов. Держите `en-US.json` присутствующим и полным: это страховочная сетка для каждого другого языка, а сырой ключ на экране — ваш первый признак того, что он неполон.
:::

## Пространство имён — где живут ваши ключи

Платформа обслуживает каждый написанный вами ключ под `plugins.<pluginId>.<key>`. Для нашего примера аддона **Shoutbox** (его `pluginId` — `pano-plugin-shoutbox` — значение, которое вы задали в `gradle.properties`, когда создавали аддон, см. [Начало работы](/ru/addon/getting-started/)), ключ под названием `widget.title` становится `plugins.pano-plugin-shoutbox.widget.title` во время поиска.

Вы никогда не пишете этот длинный префикс — ни в JSON-файлах, ни в компонентах. Внутри вашего JSON ключи остаются **без префикса** (`widget.title`), а Pano автоматически добавляет часть `plugins.<pluginId>.`. *Единственное* место, где префикс действительно появляется, — это небольшой помощник в `main.js` (показан ниже), и boilerplate уже его содержит. Это автоматическое добавление префикса — то, что удерживает ваши ключи от столкновений с ключами основной платформы или другого аддона.

## Определите свой первый ключ

Пользовательские ключи — это обычные пары ключ–значение, которые показывает ваш UI. Поместите их в `en-US.json`, вложив как угодно:

```json
{
  "widget": {
    "title": "Latest shouts",
    "empty": "No shouts yet — be the first!"
  },
  "welcome-message": "Welcome back, {username}!"
}
```

**Вложенные объекты становятся путями через точку.** `title` выше живёт внутри `widget`, поэтому вы ищете его как единственный ключ `widget.title` (не пишите буквальный ключ `"widget.title"`). Аналогично, `empty` внутри `widget` — это ключ `widget.empty`.

**Плейсхолдеры используют одинарные скобки.** `{username}` — это слот, который вы заполняете позже из своего компонента (показано в следующем разделе). Плейсхолдеры отрисовываются с одинарными скобками (`{username}`), потому что UI использует svelte-i18n. Бэкенд-шаблоны писем — единственное исключение — они используют двойные скобки; см. [одинарные против двойных скобок](#бэкенд-kotlin) ниже.

## Покажите ключ в компоненте (Svelte)

Чтобы показать перевод в компоненте, импортируйте помощник `_` из `main.js` вашего аддона — boilerplate уже его поставляет:

```svelte
<script>
  import { _ } from '../main.js';
</script>

<h2>{$_('widget.title')}</h2>
```

Три вещи, на которых спотыкается младший читатель, — все нормальные:

- **Почему он называется `_`?** `_` — это принятое в svelte-i18n имя для его функции перевода — один символ, потому что вы печатаете его постоянно.
- **Импортируете `_`, используете `$_`.** Префикс `$` — это способ Svelte читать текущее значение *хранилища* (хранилище — это реактивная коробка значений Svelte). Вы импортируете `_`; вы используете его как `$_`.
- **Поправьте `../` под глубину вашего файла.** `import { _ } from '../main.js'` предполагает, что ваш компонент сидит ровно на одну папку ниже `main.js`. В более глубоко вложенном маршруте добавьте больше `../` (например, `../../main.js`), иначе импорт вернёт 404.

`{$_('widget.title')}` ищет за вас `plugins.pano-plugin-shoutbox.widget.title`.

::: tip Контрольная точка
Компонент теперь отрисовывает **Latest shouts**. Если вы вместо этого видите сырой `plugins.pano-plugin-shoutbox.widget.title`, значит ключ отсутствует или написан с ошибкой в `en-US.json`.
:::

### Заполнение плейсхолдера

У `welcome-message` выше был слот `{username}`. Чтобы заполнить его, передайте второй аргумент — svelte-i18n называет его `options` — с вашими значениями плейсхолдеров под `values`:

```svelte
<p>{$_('welcome-message', { values: { username: 'Ada' } })}</p>
```

Это отрисовывает **Welcome back, Ada!**. Имена под `values` должны совпадать с именами `{...}` в строке. Если вы совсем забудете аргумент values, посетитель увидит на экране буквальный `{username}`.

::: tip Множественное число: «1 shout» против «5 shouts»
Не изобретайте два ключа и `if`. svelte-i18n понимает синтаксис сообщений ICU, поэтому один ключ обрабатывает оба варианта — например, `{count, plural, one {# shout} other {# shouts}}` — и вы вызываете его через `$_('shout-count', { values: { count: n } })`. См. [документацию по форматированию svelte-i18n](https://github.com/kaisermann/svelte-i18n).
:::

### Как работает помощник `_` (пропустите, если спешите)

Вам **не** нужно понимать этот сниппет, чтобы использовать `_` — `main.js` boilerplate уже его содержит, так что просто убедитесь, что он там. Но вот он:

```js
// src/main.js
import { derived } from 'svelte/store';
import { _ as i18n } from '@panomc/sdk/utils/language';

export const pluginId = 'pano-plugin-shoutbox';
export const _ = derived(i18n, ($t) => (key, options) => $t(`plugins.${pluginId}.${key}`, options));
```

*Хранилище* — это реактивная коробка значений Svelte; `derived` означает «пересчитывай всякий раз, когда меняется то, от чего я завишу» — так что когда посетитель переключает язык, каждый `$_(...)` на странице обновляется сам. `@panomc/sdk` встраивается boilerplate (см. [Разработку фронтенда](/ru/addon/frontend/)). Скопируйте это как есть. Это единственное место, где префикс `plugins.<pluginId>.` из ранее действительно написан.

::: tip Доступ к ключам вне вашего пространства имён
Если вам нужен ключ, которого *нет* под пространством имён вашего аддона, импортируйте собственное хранилище svelte-i18n — `import { _ } from '@panomc/sdk/utils/language'` — и передайте **полный путь** сами (например, `$_('plugins.pano-plugin-shoutbox.widget.title')`). `_` в вашем `main.js` — это просто это же хранилище с добавленным префиксом.
:::

## Когда мои правки появляются? Мгновенно в Режиме разработки, иначе только после пересборки

Теперь, когда у вас есть ключ на экране, вот как ваши *правки* к нему доходят до браузера. Это зависит от **Режима разработки** — переключателя, который вы включили в [Начале работы](/ru/addon/getting-started/) (**Панель → Настройки платформы**, ключ конфигурации `development-mode`). Если вы его пропустили, сначала сделайте это.

**При включённом Режиме разработки** и вашем аддоне, клонированном в папку `plugins/<pluginId>/` вашего работающего сервера Pano (папка, где работает сервер, называется *экземпляром*), Pano читает ваши `locales/*.json` **с диска в реальном времени** на каждый запрос. Отредактируйте файл, обновите (F5), и новый текст появляется — без пересборки, ровно как UI на Svelte.

**При выключенном Режиме разработки или в выпущенном jar** работающий сервер читает только копию ваших файлов локалей, упакованную *внутри jar*, — он никогда не смотрит в вашу папку исходников, поэтому редактирование файла на диске ничего не даёт, пока вы не пересоберёте jar:

```bash
./gradlew build -Pnoui
```

(`gradlew` — это скрипт-обёртка Gradle, уже находящийся в вашем проекте — запускайте его из корня проекта, что и означает ведущее `./`. `-Pnoui` пропускает пересборку UI, чтобы сэкономить время.)

::: tip Контрольная точка
Теперь вы должны увидеть jar в `build/libs/` (для локальной сборки — `pano-plugin-shoutbox-local-build.jar`). Скопируйте этот файл в папку `plugins/` вашего экземпляра Pano, заменив старый, затем **перезапустите Pano** — остановите работающий процесс и запустите его снова (см. [Начало работы](/ru/addon/getting-started/)). Отключение и повторное включение аддона в панели *не* перезагружает содержимое jar.
:::

::: tip Зачем два режима?
В продакшене локали поставляются внутри jar, чтобы аддон оставался одним самодостаточным файлом. Режим разработки меняет это на чтение с диска в реальном времени, чтобы вы могли итерировать над переводами без пересборки — по той же причине он заново обслуживает ваш UI на Svelte на каждый запрос. Полную таблицу «горячее против пересборки» см. в [Начале работы](/ru/addon/getting-started/).
:::

## Права доступа (только если ваш аддон определяет права доступа)

Если ваш аддон определяет право доступа, дайте ему человекочитаемое название и описание, чтобы администраторы могли понять его на странице **Права доступа** панели. Пропустите этот раздел, если у вашего аддона нет прав доступа.

**Определение (Kotlin):**

```kotlin
@PermissionDefinition
class ManageShoutboxPermission : PanelPermission("fa-bullhorn")
```

Строка `@PermissionDefinition` — это Kotlin-*аннотация* — маркер, который Pano сканирует, чтобы ваш класс регистрировался автоматически; копируйте её в точности (детали в [Разработке бэкенда](/ru/addon/backend/)). `fa-bullhorn` — это имя иконки Font Awesome, показываемой рядом с правом доступа в панели.

**Перевод (в вашем файле локали):**

```json
{
  "permissions": {
    "MANAGE_SHOUTBOX": {
      "title": "Manage Shoutbox",
      "description": "Allows managing shouts shown on the home page."
    }
  }
}
```

Вы не выбираете ключ `MANAGE_SHOUTBOX` — он выводится из имени класса по фиксированному правилу: отбросьте суффикс `Permission`, затем преобразуйте в `UPPER_SNAKE_CASE`.

- `ManageShoutboxPermission` → `MANAGE_SHOUTBOX`

::: tip Контрольная точка
Откройте **Панель → Права доступа**. Ваша запись теперь должна читаться как **Manage Shoutbox** со своим описанием, вместо сырого ключа `MANAGE_SHOUTBOX`.
:::

## Журналы активности (только если ваш аддон записывает активность)

Если ваш аддон записывает административные действия, определите шаблон для каждого из них под `activity-logs`. Эти строки появляются на странице **Активность** панели. Пропустите этот раздел, если ваш аддон не записывает активность.

**Определение (Kotlin):**

```kotlin
class CreatedShoutLog(
    userId: Long,
    username: String,
    pluginId: String,
    message: String,
) : PluginActivityLog(
    userId = userId,
    pluginId = pluginId,
    details = JsonObject().put("target", message).put("username", username),
)
```

Для перевода важна только строка `details`: каждый ключ, который вы в неё `put`, — здесь `target` и `username` — становится `{placeholder}`, который вы можете использовать в тексте. Остальное (`userId`, `pluginId`, конструкторная обвязка) — это бэкенд-связывание, описанное в [Разработке бэкенда](/ru/addon/backend/).

**Перевод (в вашем файле локали):**

```json
{
  "activity-logs": {
    "CREATED_SHOUT": "<b>{username}</b> posted a shout: {target}."
  }
}
```

Ключ следует тому же правилу, что и права доступа, но отбрасывает суффикс `Log`:

- `CreatedShoutLog` → `CREATED_SHOUT`

Плейсхолдеры `{username}` и `{target}` заполняются из JSON `details` журнала — класс Kotlin поставляет *данные*, а эта запись локали поставляет *формулировку*.

::: tip Можно ли использовать HTML вроде `<b>` в моих строках?
Здесь — да: `<b>...</b>` работает, потому что страница Активность отрисовывает эти конкретные строки как HTML. Это **не** общее разрешение помещать HTML в каждую строку локали. В обычных пользовательских ключах, показываемых через `{$_(...)}`, svelte-i18n возвращает сырую строку, а Svelte её экранирует, поэтому `<b>` появился бы на экране как буквальный текст `<b>`. Держите HTML-разметку только в строках журнала активности.
:::

::: tip Контрольная точка
Выполните действие, затем откройте **Панель → Активность**. Отрисованное предложение должно появиться с именем пользователя жирным.
:::

## Один полный `en-US.json`

Примеры выше были фрагментами. Вот как они складываются в единый файл. **`permissions` и `activity-logs` сидят в корне**, рядом с вашими пользовательскими ключами — никогда не вложенными внутрь другого объекта. Pano читает эти два раздела напрямую, чтобы связать их с основными страницами Права доступа и Активность.

```json
{
  "widget": {
    "title": "Latest shouts",
    "empty": "No shouts yet — be the first!"
  },
  "welcome-message": "Welcome back, {username}!",
  "permissions": {
    "MANAGE_SHOUTBOX": {
      "title": "Manage Shoutbox",
      "description": "Allows managing shouts shown on the home page."
    }
  },
  "activity-logs": {
    "CREATED_SHOUT": "<b>{username}</b> posted a shout: {target}."
  }
}
```

## Бэкенд (Kotlin)

Две бэкенд-системы — и *только* эти две — читают ваши файлы локалей. Ваш собственный Kotlin-код никогда не ищет перевод напрямую; если вам нужен текст на бэкенде, он течёт через одну из них:

- **Журналы активности.** Когда ваш аддон записывает административное действие, вы вставляете подкласс `PluginActivityLog` (см. [Разработку бэкенда](/ru/addon/backend/)). Страница Активность отрисовывает его, ища `activity-logs.<KEY>` и заполняя плейсхолдеры `{...}` из нагрузки `details` журнала. Локализованный текст живёт *здесь*, в файле локали — никогда в Kotlin.
- **Почта.** Если ваш аддон отправляет письма, тема и тело приходят из шаблонов **Handlebars**. (Handlebars — это язык шаблонов; `{{name}}` помечает переменную для заполнения.) Эти шаблоны **не** являются частью ваших `locales/*.json` — они живут рядом с почтовым кодом вашего аддона, а не в файлах локалей. Аддон `pano-plugin-auth-guard` — образец для отправки шаблонных писем (см. [Разработку бэкенда](/ru/addon/backend/)).

::: warning Одинарные против двойных скобок
Всё, что отрисовывает UI, — пользовательские ключи, названия прав доступа, строки журнала активности — проходит через svelte-i18n и интерполируется с **одинарными** скобками: `{username}`. **Единственное** место, где вы пишете **двойные** скобки `{{variable}}`, — это внутри бэкенд-шаблонов писем (Handlebars). Использование не того стиля оставляет сырые `{...}` видимыми на экране.
:::

## Переопределения администратора и новые языки

Одна из приятных мелочей Pano: администраторы могут редактировать переводы вашего аддона прямо из панели.

- **Переопределения:** администратор может изменить любой определённый вами ключ — новая формулировка без касания вашего jar.
- **Новые языки:** администратор может добавить язык, который ваш аддон не поставляет, переведя ваши ключи в панели.

Эти правки хранятся Pano отдельно и переживают обновления аддона. **Чей текст побеждает?** Переопределение администратора всегда бьёт текст в вашем jar. Так что если вы измените собственную формулировку в новом релизе, кастомизированное значение администратора остаётся в силе для тех ключей, которые он переопределил; ваше новое значение по умолчанию показывается только для ключей, которых администратор никогда не касался. Формулировка владельца сайта никогда не перезаписывается молча обновлением.

## Если что-то выглядит не так

| Вы видите | Вероятная причина | Исправление |
|---|---|---|
| Сырой путь ключа на экране (`plugins.pano-plugin-shoutbox.widget.title`) | Ключ отсутствует или написан с ошибкой, в языке посетителя *и* в `en-US.json` | Добавьте или исправьте ключ в `en-US.json` (запасной вариант) |
| Буквальный `{username}` на экране | Вы не передали значение для плейсхолдера | Передайте его: `$_('welcome-message', { values: { username: ... } })` |
| Буквальный `{{name}}` на экране | Неверный стиль скобок для отрисовщика | UI использует одинарные `{ }`; только почта/Handlebars использует двойные `{{ }}` |
| Правки файла локали не появляются после обновления | Режим разработки выключен или вы запускаете выпущенный jar | Включите Режим разработки или пересоберите jar и перезапустите Pano |
| Турецкий/русский текст показывается как `Ã¶`-подобная тарабарщина | Файл был сохранён в неверной кодировке | Пересохраните файл как UTF-8 |

::: warning Сломанный JSON-файл ломает все свои ключи, а не один
JSON строг: единственная завершающая запятая, случайный комментарий или незакрытая скобка делают *весь файл* невалидным. Когда файл нельзя разобрать, ни один из его ключей не разрешается, поэтому вы увидите сырые пути ключей там, где должны быть эти строки, — часто по всей странице сразу. Если страница внезапно показывает пути ключей повсюду, сначала подозревайте синтаксическую ошибку JSON и прогоните файл через любой валидатор JSON.
:::
