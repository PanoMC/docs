# Сборка и публикация

Эта страница проведёт вас от аддона, который работает на вашей машине, до аддона, который могут установить владельцы серверов. Вы сделаете одну релизную сборку, узнаете, как версии выбираются за вас, и опубликуете на официальном Маркетплейсе Pano на panomc.com — бесплатно или (с чуть большим количеством связывания) как платный аддон.

«Публикация» здесь означает превращение вашего аддона в **релизный jar** (`.jar` — единственный zip-подобный файл, который держит весь ваш скомпилированный аддон) и размещение его там, где другие могут его получить.

Примеры на этой странице используют аддон **Shoutbox** из [Начала работы](/ru/addon/getting-started/) — небольшой аддон, чей ID плагина — `pano-plugin-shoutbox`. Везде, где вы видите `shoutbox` или `pano-plugin-shoutbox`, подставляйте собственный `pluginId` вашего аддона.

Если вы ещё не собрали свой аддон, начните с [Начала работы](/ru/addon/getting-started/) и [Разработки бэкенда](/ru/addon/backend/).

## Прежде чем начать

Приготовьте эти четыре вещи. Остальная часть страницы предполагает их:

- **Ваш аддон уже собирается локально** — запуск `./gradlew build` в папке вашего аддона завершается без ошибок. Если нет, сначала вернитесь к [Началу работы](/ru/addon/getting-started/) и [Разработке бэкенда](/ru/addon/backend/).
- **Код вашего аддона живёт в репозитории GitHub.**
- **GitHub Actions включён** для этого репозитория (он включён по умолчанию для новых репозиториев).
- **У вас есть бесплатный аккаунт на [panomc.com](https://panomc.com).**

## Первый чек-лист релиза

Остальная часть страницы объясняет каждую часть подробно, но вот вся работа по порядку, чтобы вы всегда знали, что осталось. Каждый шаг ссылается на своё полное объяснение и заканчивается тем, как понять, что он сработал.

Большая часть работы — это одноразовая настройка (шаги 2–6). После этого каждый будущий релиз — это просто шаг 7 — закоммитить и запушить.

1. **Соберите один раз локально**, чтобы убедиться, что jar производится. → [Релизная сборка](#релизная-сборка). *Готово, когда вы видите `BUILD SUCCESSFUL` и jar в `build/libs/`.*
2. **Создайте свой ресурс на Маркетплейсе** на panomc.com. → [Создайте ресурс](#_1-создаите-ресурс). *Готово, когда у вашего аддона есть страница на panomc.com с пустым списком версий.*
3. **Создайте API-токен** и сохраните его как секрет репозитория с именем `PANO_PROD_TOKEN` (для ветки `main`). → [Создайте API-токен](#_2-создаите-api-токен). *Готово, когда секрет перечислен в настройках вашего репозитория.*
4. **Создайте секрет `TOKEN_GITHUB`** (GitHub Personal Access Token). Пропустите это — и каждый релиз проваливается на самом первом шаге. → [Обязательно: секрет `TOKEN_GITHUB`](#обязательно-секрет-token-github).
5. **Добавьте `.releaserc.json`** с плагином публикации Pano и замените в нём плейсхолдеры. → [Разбор `.releaserc.json`](#разбор-releaserc-json).
6. **Добавьте один шаг установки в workflow, в обе задачи**, чтобы плагин публикации Pano был доступен. Пропустите это — и релиз провалится с `Cannot find module @PanoMC/semantic-release-pano`. → [Плагин Pano нужно установить в workflow](#плагин-pano-нужно-установить-в-workflow).
7. **Закоммитьте с сообщением в формате [conventional-commit](https://www.conventionalcommits.org/) и запушьте** в `main`. → [Отправьте push и наблюдайте за публикацией](#_3-отправьте-push-и-наблюдаите-за-публикациеи). *Готово, когда прогон Actions показывает зелёную галочку и новая версия появляется на вашем ресурсе.*

## Релизная сборка

Релизная сборка компилирует ваш бэкенд на Kotlin **и** собирает и встраивает UI на Svelte в jar (`.jar` — единственный самодостаточный файл, который вы отправляете). Это обычная:

```bash
./gradlew build
```

Запускайте это из корневой папки вашего аддона. Gradle — это инструмент сборки для стороны Kotlin — считайте `./gradlew build` эквивалентом `npm run build` в мире JVM, а `./gradlew` («обёртка Gradle») — скриптом в вашем проекте, который запускает нужную версию Gradle за вас. Вы должны увидеть `BUILD SUCCESSFUL` через минуту-две.

Вывод оказывается в:

```
build/libs/pano-plugin-shoutbox-<version>.jar
```

Локально `<version>` всегда равна `local-build` (так что файл — это `pano-plugin-shoutbox-local-build.jar`). Теперь вы должны увидеть этот файл внутри `build/libs/`. Настоящие номера версий приходят из CI — **CI** (непрерывная интеграция) — это автоматическая сборка, которую GitHub запускает на своих собственных серверах каждый раз, когда вы делаете push. См. [Версионирование](#версионирование) ниже.

::: warning Релизным jar нужен UI
**Для релиза всегда запускайте обычную `./gradlew build` — никогда не добавляйте `-Pnoui`.** Флаг `-Pnoui` пропускает сборку UI на Bun/rollup, и это может отправить сломанный аддон двумя разными способами:

- **Если вы никогда не собирали UI:** jar отправляется **вообще без UI** — ваш аддон загружается без экранов панели или темы.
- **Если старая полная сборка оставила `plugin-ui.zip`:** jar молча вшивает этот **устаревший** UI, так что вы отправляете устаревший интерфейс, не замечая этого.

Если хотите быть уверены, что старый zip UI не переиспользуется, запустите `./gradlew clean build` (задача `clean` сначала удаляет предыдущий вывод сборки). `-Pnoui` предназначен только для быстрого бэкенд-только цикла разработки (см. [Начало работы](/ru/addon/getting-started/#цикл-разработки)).
:::

jar полностью самодостаточен: бэкенд на Kotlin, встроенный бандл UI, локали и `logo.png` — всё живёт внутри него. Больше нечего отправлять.

## Версионирование

У каждого релиза есть номер версии (вроде `1.0.0`), чтобы серверы знали, когда доступно обновление.

Вы **не** повышаете версию вручную. Версии решаются из ваших сообщений коммитов, которые должны следовать [Conventional Commits](https://www.conventionalcommits.org/) — простому формату, где каждый коммит начинается со слова вроде `feat:` (новая возможность), `fix:` (исправление бага) или `chore:` (уборка). Эти слова управляют и следующим номером версии, и генерируемым changelog. В версии вроде `1.2.3` `1` — это **major**, `2` — **minor**, а `3` — **patch**. `feat:` повышает minor-версию, `fix:` повышает patch-версию, а `feat:` с футером `BREAKING CHANGE:` повышает major-версию.

Четыре разных имени в этом проекте звучат похоже, но означают разные вещи. Вот таблица, которую стоит держать в порядке:

| Имя | Живёт в | Кто задаёт | Что означает |
|---|---|---|---|
| `version` | `gradle.properties` | CI, во время релиза | **Собственная версия вашего аддона** (вроде `1.1.0`). Остаётся `local-build` в вашей рабочей копии; никогда не редактируйте её вручную. |
| `pluginPanoVersion` | `gradle.properties` | Вы (оставьте на `local-build`) | Копируется в манифест jar. CI её **не** трогает. |
| `pano-version` | манифест jar | приходит прямо из `pluginPanoVersion` | Просто вшитая копия `pluginPanoVersion` внутри собранного jar. |
| `panoVersion` | `.releaserc.json` | Вы | Версия платформы Pano, показываемая на вашем листинге **Маркетплейса** (задаётся в файле конфигурации ниже). |

::: warning Не редактируйте версии вручную
Оставьте и `version`, и `pluginPanoVersion` в `gradle.properties` на `local-build`. Во время релиза CI внедряет настоящую `version` (через `-Pversion` — способ, которым Gradle принимает значение из командной строки, `-Pversion=1.1.0`) из вашей истории коммитов. Ручное повышение её или редактирование тега (git-тега, который semantic-release создаёт для каждого релиза) ломает автоматизацию. CI **не** внедряет `pluginPanoVersion`; атрибут `pano-version` манифеста остаётся `local-build`. Версия Pano, *объявляемая на Маркетплейсе*, — это отдельное значение, задаваемое опцией `panoVersion` в `.releaserc.json` (ниже). Пусть ваши сообщения коммитов управляют версией. Детали см. в [Конфигурации манифеста](/ru/addon/manifest/).
:::

## Каналы релизов

Boilerplate настроен на два канала релизов, определяемых тем, в какую ветку вы делаете push:

| Ветка | Канал | Версия выглядит как |
|---|---|---|
| `dev` | Предрелиз | `1.1.0-dev.3` |
| `main` | Стабильный | `1.1.0` |

**Предрелиз** — это тестовая сборка, которую вы можете попробовать до настоящего релиза. Суффикс `-dev.3` означает «3-я dev-сборка на пути к `1.1.0`». Владельцы серверов, устанавливающие стабильный канал, никогда не видят их; предрелиз — для того, чтобы вы могли безопасно отрепетировать релиз.

::: tip Что такое GitHub Actions (прочитайте это сначала)
**GitHub Actions** запускает скрипты на собственных серверах GitHub всякий раз, когда вы делаете push. Скрипт — это YAML-файл в вашем репозитории — здесь `.github/workflows/release.yml`. Один файл содержит одну или несколько **задач (jobs)**; каждая задача — это список **шагов (steps)**; шаг выполняет команду или готовое действие. Этот файл и делает всё ниже: собирает ваш jar и публикует его.
:::

Push в `dev` или `main` запускает workflow GitHub Actions, поставляемый с `pano-boilerplate-plugin` по адресу `.github/workflows/release.yml`. Этот workflow вычисляет следующую версию из ваших коммитов, запускает `./gradlew build` (настоящую сборку UI — не `-Pnoui`), а затем запускает **semantic-release** — инструмент, который читает ваши сообщения коммитов, решает следующий номер версии, пишет changelog и публикует релиз, всё автоматически.

**Одна правка обязательна для публикации на Маркетплейсе.** «Из коробки» workflow boilerplate запускает semantic-release *без* установленного плагина публикации Pano. Чтобы также публиковать на Маркетплейсе, вы должны добавить один шаг установки в workflow — это расписано под [Плагин Pano нужно установить в workflow](#плагин-pano-нужно-установить-в-workflow) ниже, и это шаг 6 [чек-листа](#первыи-чек-лист-релиза). Помимо этой одной правки и секретов, которые она ожидает, вы не трогаете workflow для обычного аддона.

Триггер вверху этого workflow — просто:

```yaml
on:
  push:
    branches: ['dev', 'main']
```

Это уже в boilerplate — вы это не меняете. Показано, чтобы вы знали, что **push в `dev` или `main` — это весь триггер релиза**: отдельной кнопки «опубликовать» нет.

## Разбор `.releaserc.json`

`.releaserc.json` — это место, где настраивается semantic-release. Этот файл упоминает API-токен и ресурс Маркетплейса, которые вы создаёте в [следующем разделе](#публикация-на-официальном-маркетплеисе-pano) — это нормально. Прочитайте этот разбор, чтобы понять файл, настройте его, затем создайте эти вещи потом.

Boilerplate поставляет `.releaserc.json`, который создаёт релиз GitHub с прикреплённым jar. Чтобы также публиковать на Маркетплейсе, вы добавляете плагин `@PanoMC/semantic-release-pano`.

Три имени здесь содержат слово «release» — держите их порознь:

- **`semantic-release`** — сам инструмент.
- **`@semantic-release/commit-analyzer`, `@semantic-release/release-notes-generator`, `@semantic-release/github`** — стандартные, готовые плагины для этого инструмента.
- **`@PanoMC/semantic-release-pano`** — единственный специфичный для Pano плагин, который вы добавляете и который делает публикацию на Маркетплейсе.

В списке `plugins` ниже каждая запись — это либо имя плагина само по себе, либо пара `[name, options]` — массив из двух элементов: имя плагина плюс его настройки. Вот полный файл для Shoutbox — адаптированный 1:1 из реального конфига релиза `pano-plugin-faq`:

```json
{
  "branches": [
    { "name": "dev", "prerelease": true },
    "main"
  ],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/github", {
      "assets": [
        { "path": "build/libs/*.jar", "label": false },
        { "path": "LICENSE", "label": false }
      ]
    }],
    ["@PanoMC/semantic-release-pano", {
      "file": "build/libs/pano-plugin-shoutbox-${version}.jar",
      "panoVersion": "1.0.0",
      "useGitHubLink": true,
      "repositoryUrl": "https://github.com/YourName/pano-plugin-shoutbox.git",
      "configs": [
        {
          "resourceId": "pano-plugin-shoutbox",
          "panoUrl": "https://api-dev.panomc.com",
          "tokenVar": "PANO_TOKEN",
          "branches": ["dev"]
        },
        {
          "resourceId": "pano-plugin-shoutbox",
          "panoUrl": "https://api.panomc.com",
          "tokenVar": "PANO_PROD_TOKEN",
          "branches": ["main"]
        }
      ]
    }]
  ],
  "repositoryUrl": "https://github.com/YourName/pano-plugin-shoutbox.git"
}
```

::: tip Замените эти плейсхолдеры
Прежде чем этот файл заработает, замените примерные значения на свои собственные:

- `YourName` (**2 места** — обе строки `repositoryUrl`) → ваше имя пользователя или организация на GitHub.
- `pano-plugin-shoutbox` (**5 мест** — путь `file`, оба значения `resourceId` и внутри обеих строк `repositoryUrl`) → ваш собственный `pluginId`.

Путь `file` должен содержать ваш `pluginId` **в точности**, иначе релиз провалится гораздо позже с непонятной ошибкой «file not found». И да, `repositoryUrl` появляется **дважды** намеренно (один раз внутри опций плагина Pano, один раз на верхнем уровне) — задайте URL своего репозитория в **обоих**.
:::

Поле за полем:

| Поле | Что оно делает |
|---|---|
| `commit-analyzer` и `release-notes-generator` | Первые две записи в `plugins`. Стандартные плагины semantic-release — оставьте их ровно как есть. |
| `@semantic-release/github` `assets` | Прикрепляет ваш `build/libs/*.jar` (и `LICENSE`) к релизу GitHub. **Держите плагины в этом порядке — плагин GitHub должен выполниться до плагина Pano**, чтобы jar был уже прикреплён, когда `useGitHubLink` его понадобится. (`"label": false` просто оставляет собственное имя каждого файла как его ярлык для скачивания — не трогайте.) |
| `file` | Путь к собранному jar. `${version}` подставляется версией релиза. |
| `panoVersion` | Версия платформы Pano, под которую ваш аддон собран и протестирован (например, `1.0.0`) — это значение, показываемое на вашем листинге Маркетплейса. Это **не** собственная версия вашего аддона. |
| `useGitHubLink` | `true` = не перезагружать jar; вместо этого указать Маркетплейсу на jar, уже прикреплённый к релизу GitHub (плюс его SHA-256-хеш — отпечаток, чтобы скачивание можно было проверить). Идеально для бесплатных аддонов — без дублирующей загрузки. Премиум-аддоны вместо этого загружают jar напрямую (премиум: установите `useGitHubLink: false`; см. [Премиум-аддоны и лицензирование](/ru/addon/premium/)). |
| `configs[]` | По одной записи на канал. Каждая говорит, **на какой Маркетплейс публиковать** и **с каким токеном**, ограниченная по `branches`. |
| `resourceId` | Ваш ресурс на Маркетплейсе — для аддонов это ваш `pluginId` (см. ниже). |
| `panoUrl` | API Маркетплейса: `https://api-dev.panomc.com` для канала `dev`, `https://api.panomc.com` для `main`. |
| `tokenVar` | Имя секрета GitHub, хранящего API-токен: `PANO_TOKEN` для dev, `PANO_PROD_TOKEN` для продакшена. |
| `branches` | Ограничивает конфиг одним каналом, чтобы push в `dev` никогда не касался продакшен-Маркетплейса (и отсутствующий `PANO_PROD_TOKEN` не проваливал `dev`-сборку). |

::: tip Почему два `configs` — и что такое «dev-Маркетплейс»
`api-dev.panomc.com` — это отдельный **песочный** Маркетплейс: у него собственные ресурсы, собственные токены и собственный вход, и ничто из опубликованного там никогда не видят реальные владельцы серверов. Разделение по ветке означает, что вы можете отрепетировать релиз на `api-dev.panomc.com` из своей ветки `dev`, а затем отправить ровно тот же код на реальный `api.panomc.com` из `main`.

**Если вы публикуете свой первый аддон, держите это простым: публикуйте только в `main`.** Удалите запись конфига `dev` (и ветку `{ "name": "dev", "prerelease": true }`), и вам нужен только один токен, `PANO_PROD_TOKEN`. Добавьте dev-песочницу позже, если когда-нибудь захотите канал для репетиций.
:::

### Плагин Pano нужно установить в workflow

`@PanoMC/semantic-release-pano` **не** опубликован в npm и не является зависимостью boilerplate, поэтому перечисления его в `.releaserc.json` само по себе недостаточно — semantic-release провалится с *«Cannot find module @PanoMC/semantic-release-pano»*.

::: warning Это единственная обязательная правка workflow
Вы должны добавить этот шаг установки в **обе** задачи в `.github/workflows/release.yml` — задачу сухого прогона версии (`get-next-version`) и задачу релиза (`build-and-release`) — размещённый **перед** шагом `semantic-release` каждой задачи:

```yaml
- run: npm install -D git+https://github.com/PanoMC/semantic-release-pano.git
```

(`-D` = установить как dev-зависимость; URL `git+` устанавливает плагин прямо с GitHub, потому что его нет в npm. Этот один шаг использует `npm`/`npx`, хотя остальной проект использует Bun — здесь так и должно быть.)

Например, в задаче релиза он идёт прямо перед существующим шагом `Release`:

```yaml
      # add this line...
      - run: npm install -D git+https://github.com/PanoMC/semantic-release-pano.git

      # ...before the step already in the file:
      - name: Release
        env:
          GITHUB_TOKEN: ${{ secrets.TOKEN_GITHUB }}
        run: npx semantic-release@24.2.6
```

Сделайте то же в задаче сухого прогона, перед её шагом `npx semantic-release --dry-run`. Если добавите это только в одну задачу, прогон провалится в другой с `Cannot find module @PanoMC/semantic-release-pano` — эта ошибка означает, что вы пропустили одну из двух задач. У workflow `pano-plugin-faq`, из которого скопирован этот конфиг, есть ровно этот шаг.
:::

## Публикация на официальном Маркетплейсе Pano

Маркетплейс на [panomc.com](https://panomc.com) — это место, где владельцы серверов находят и устанавливают аддоны прямо из своей панели. **Ресурс** — это листинг вашего аддона в магазине на panomc.com. Публикация занимает три шага: создать ресурс, создать API-токен, затем позволить автоматизации загружать версии. (Секрет `TOKEN_GITHUB` тоже обязателен — см. врезку после шага 2.)

### 1. Создайте ресурс

1. Зарегистрируйтесь (или войдите) на **panomc.com**.
2. Из области своего профиля откройте **Create Resource** и выберите тип **Plugin**.
3. Выберите категорию, заполните название и описание.
4. Выберите ценообразование: **бесплатно** или **платно**. Если вы просто публикуете бесплатный аддон, выберите **бесплатно** — это всё, о чём эта страница. Выбор **платно** добавляет шаг лицензии, описанный в [Премиум-аддоны и лицензирование](/ru/addon/premium/).

Теперь вы должны увидеть страницу вашего аддона на panomc.com с **пустым списком версий** — автоматизация заполнит его, когда вы сделаете push.

::: tip ID ресурса вашего аддона — это ID вашего плагина
`resourceId` вашего аддона на Маркетплейсе — это **ровно ваш `pluginId`** — для Shoutbox это `pano-plugin-shoutbox`. Вот почему `configs[]` выше используют `"resourceId": "pano-plugin-shoutbox"`, а не случайный ID. (Темы устроены иначе — они используют случайный UUID — а аддоны нет.) `pluginId` — это единственная идентичность, которую Pano использует везде, где ваш аддон касается системы; полный список — на [Конфигурации манифеста](/ru/addon/manifest/).
:::

### 2. Создайте API-токен

1. На panomc.com откройте **Profile → Settings → API Tokens** и нажмите **Create**.
2. Скопируйте токен немедленно — он показывается **только один раз**, в модальном окне сразу после создания.
3. В своём репозитории GitHub перейдите в **Settings → Secrets and variables → Actions** и добавьте токен как **секрет репозитория** (зашифрованное значение, которое могут прочитать только ваши прогоны Actions), названный в соответствии с вашим `tokenVar`:
   - `PANO_PROD_TOKEN` для канала `main` (продакшен) — тот, что настраивают первым.
   - `PANO_TOKEN` для канала `dev`, **только если** вы сохранили конфиг dev-песочницы. Это *другой* токен, созданный в отдельной песочнице `api-dev.panomc.com`, — не то же значение, что `PANO_PROD_TOKEN`.

Страница вашего репозитория **Settings → Secrets and variables → Actions** теперь должна перечислять `PANO_PROD_TOKEN` (и `PANO_TOKEN` тоже, если вы публикуете в `dev`).

::: warning Никогда не коммитьте токен
API-токен даёт права на публикацию вашего ресурса. Храните его только как секрет GitHub (или локальную переменную окружения). Никогда не помещайте его в `.releaserc.json`, коммит или любой файл в репозитории.
:::

### Обязательно: секрет `TOKEN_GITHUB`

Самому релизу GitHub нужен второй секрет, `TOKEN_GITHUB`, который workflow boilerplate читает (как `secrets.TOKEN_GITHUB`) в нескольких местах, включая сухой прогон версии — репетиционную задачу, вычисляющую следующую версию, ничего не публикуя. Встроенный `GITHUB_TOKEN` GitHub **не** предоставляется под этим именем, поэтому вы должны создать его сами.

::: warning Без этого каждый релиз проваливается на самом первом шаге
Создайте `TOKEN_GITHUB` как **Personal Access Token (PAT)** — токен, который вы генерируете под своим собственным аккаунтом GitHub:

1. На GitHub перейдите в аватар → **Settings → Developer settings → Personal access tokens**.
2. Создайте **classic**-токен со скоупом **`repo`** (semantic-release нужен он, чтобы создавать релизы и пушить теги).
3. Скопируйте его, затем добавьте в свой репозиторий под **Settings → Secrets and variables → Actions** как секрет с именем ровно **`TOKEN_GITHUB`**.

Список секретов вашего репозитория теперь должен показывать `TOKEN_GITHUB`.
:::

::: tip Предпочитаете встроенный токен GitHub?
Если вы предпочли бы не создавать PAT, вы можете отредактировать workflow, чтобы он читал `secrets.GITHUB_TOKEN` (автоматический токен GitHub на каждый прогон) вместо `secrets.TOKEN_GITHUB`. Путь с PAT выше — это значение по умолчанию boilerplate и не требует правки workflow, поэтому это рекомендуемый путь.
:::

### 3. Отправьте push и наблюдайте за публикацией

С созданным ресурсом, добавленными обоими обязательными секретами и `.releaserc.json` (плюс шагом установки в workflow) на месте, публикация — это просто коммит с сообщением в формате [conventional-commit](https://www.conventionalcommits.org/) и push:

```bash
git push origin main    # or dev, if you kept the dev sandbox
```

Теперь откройте вкладку **Actions** вашего репозитория. Прогон, названный по имени workflow (**Pano Plugin Build**), должен появиться в течение минуты. Обычно он завершается за несколько минут.

- **Зелёная галочка** = выпущено. Откройте свой ресурс на panomc.com — новая версия должна быть в списке, и владельцы серверов увидят обновление в своей панели под **Панель → Аддоны**.
- **Красный крест** = что-то провалилось. Кликните в проваленный шаг, чтобы прочитать ошибку. Две самые частые:
  - `Cannot find module @PanoMC/semantic-release-pano` — вы пропустили шаг установки в одной из двух задач (шаг 6 / [выноска про установку](#плагин-pano-нужно-установить-в-workflow)).
  - Сбой авторизации на самом первом шаге — отсутствующий или неверный секрет `TOKEN_GITHUB` (врезка [`TOKEN_GITHUB`](#обязательно-секрет-token-github) выше).

## Ручное распространение

Вы не обязаны использовать Маркетплейс. Поскольку релизный jar полностью самодостаточен, любой может установить его напрямую:

- Прикрепите jar к релизу GitHub и поделитесь ссылкой, или
- Передайте `.jar` владельцу сервера для загрузки через **Панель → Аддоны → Загрузить**.

Разница, которую видит пользователь: аддоны, скачанные с Маркетплейса, помечены как **проверенные (verified)**, тогда как загруженные вручную jar-файлы — нет. Для публичного аддона Маркетплейс сильно предпочтительнее — он даёт вам доставку обновлений, страницу в магазине и значок verified бесплатно.
