# Сборка и публикация

Как только ваш аддон работает локально, следующий шаг — превратить его в релизный jar и показать его владельцам серверов. Эта страница охватывает релизную сборку, то, как версии определяются за вас, и то, как публиковаться на официальном Маркетплейсе Pano на panomc.com — бесплатно или (с чуть большим количеством настройки) как премиум-аддон.

Если вы ещё не собрали свой аддон, начните с [Начала работы](/ru/addon/getting-started/) и [Разработки бэкенда](/ru/addon/backend/).

## Релизная сборка

Релизная сборка компилирует ваш бэкенд на Kotlin **и** собирает и встраивает UI на Svelte в jar. Это простая команда:

```bash
./gradlew build
```

Результат оказывается в:

```
build/libs/pano-plugin-shoutbox-<version>.jar
```

Локально `<version>` всегда равна `local-build` (так что файл — это `pano-plugin-shoutbox-local-build.jar`). Настоящие номера версий приходят из CI — см. [Версионирование](#версионирование) ниже.

::: warning Релизным jar нужен UI
Никогда не используйте `./gradlew build -Pnoui` для релиза. Флаг `-Pnoui` пропускает сборку UI на Bun/rollup. Если `plugin-ui.zip` никогда не собирался, jar поставляется **без** какого-либо UI, и ваш аддон загружается вообще без экранов в панели и теме. Хуже того, если предыдущая полная сборка оставила `src/main/resources/plugin-ui.zip` (он в gitignore, но остаётся в вашем рабочем дереве), сборка с `-Pnoui` вшивает этот **устаревший** zip в jar — молча поставляя устаревший UI, который легко не заметить. В любом случае всегда запускайте чистую полную `./gradlew build` для релизов; `-Pnoui` предназначен только для быстрого бэкенд-только цикла разработки (см. [Начало работы](/ru/addon/getting-started/#цикл-разработки)).
:::

Jar полностью самодостаточен: бэкенд на Kotlin, встроенный UI-бандл, локали и `logo.png` — всё живёт внутри него. Больше нечего поставлять.

## Версионирование

У каждого релиза есть номер версии (вроде `1.0.0`), чтобы серверы знали, когда доступно обновление.

Вы **не** повышаете версию вручную. Версии определяются из ваших сообщений коммитов, которые должны следовать [Conventional Commits](https://www.conventionalcommits.org/) — простому формату, где каждый коммит начинается со слова вроде `feat:` (новая возможность), `fix:` (исправление ошибки) или `chore:` (обслуживание). Эти слова управляют и следующим номером версии, и генерируемым списком изменений. `feat:` повышает минорную версию, `fix:` повышает патч-версию, а `feat:` с футером `BREAKING CHANGE:` повышает мажорную версию.

::: warning Не редактируйте версии вручную
Оставьте и `version`, и `pluginPanoVersion` в `gradle.properties` равными `local-build`. Во время релиза CI внедряет настоящую `version` (через `-Pversion`) из истории ваших коммитов — повышение вручную или редактирование тега ломает автоматизацию. CI **не** внедряет `pluginPanoVersion`; атрибут манифеста `pano-version` остаётся `local-build`. Версия Pano, *анонсируемая на Маркетплейсе*, — это отдельное значение, задаваемое опцией `panoVersion` в `.releaserc.json` (ниже). Пусть версией управляют ваши сообщения коммитов. Подробности см. в [Конфигурации манифеста](/ru/addon/manifest/).
:::

## Каналы выпуска

Boilerplate настроен на два канала выпуска, определяемых тем, в какую ветку вы пушите:

| Ветка | Канал | Как выглядит версия |
|---|---|---|
| `dev` | Предрелиз | `1.1.0-dev.3` |
| `main` | Стабильный | `1.1.0` |

Push в любую ветку запускает рабочий процесс GitHub Actions, который поставляется с `pano-boilerplate-plugin` по пути `.github/workflows/release.yml`. Этот рабочий процесс вычисляет следующую версию из ваших коммитов, запускает `./gradlew build` (настоящую сборку UI — не `-Pnoui`), а затем запускает semantic-release, который создаёт релиз GitHub. Рабочий процесс boilerplate запускает **голый** `semantic-release`, поэтому, чтобы также публиковаться на Маркетплейсе, вы должны добавить в рабочий процесс один шаг, устанавливающий плагин публикации Pano — см. выноску в следующем разделе. Кроме этой одной правки и секретов, которые она ожидает, вы не трогаете рабочий процесс для обычного аддона.

Триггер вверху этого рабочего процесса просто:

```yaml
on:
  push:
    branches: ['dev', 'main']
```

## Разбор `.releaserc.json`

`.releaserc.json` — это место, где настраивается semantic-release. Boilerplate поставляет такой, что создаёт релиз GitHub с прикреплённым jar. Чтобы также публиковаться на Маркетплейсе, добавьте плагин `@PanoMC/semantic-release-pano`. Вот полный файл для Shoutbox — адаптированный один в один из настоящего конфига релиза `pano-plugin-faq`:

```json
{
  "branches": [
    { "name": "dev", "prerelease": true },
    "main"
  ],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/github", {
      "assets": [
        { "path": "build/libs/*.jar", "label": false },
        { "path": "LICENSE", "label": false }
      ]
    }],
    ["@PanoMC/semantic-release-pano", {
      "file": "build/libs/pano-plugin-shoutbox-${version}.jar",
      "panoVersion": "1.0.0",
      "useGitHubLink": true,
      "repositoryUrl": "https://github.com/YourName/pano-plugin-shoutbox.git",
      "configs": [
        {
          "resourceId": "pano-plugin-shoutbox",
          "panoUrl": "https://api-dev.panomc.com",
          "tokenVar": "PANO_TOKEN",
          "branches": ["dev"]
        },
        {
          "resourceId": "pano-plugin-shoutbox",
          "panoUrl": "https://api.panomc.com",
          "tokenVar": "PANO_PROD_TOKEN",
          "branches": ["main"]
        }
      ]
    }]
  ],
  "repositoryUrl": "https://github.com/YourName/pano-plugin-shoutbox.git"
}
```

::: warning Плагин Pano должен быть установлен в рабочем процессе
`@PanoMC/semantic-release-pano` **не** опубликован в npm и не является зависимостью boilerplate, поэтому одного лишь перечисления его в `.releaserc.json` недостаточно — semantic-release упадёт с *«Cannot find module @PanoMC/semantic-release-pano»*. Добавьте шаг установки в **оба** задания в `.github/workflows/release.yml` (задание сухого прогона версии и задание релиза), перед их шагом `semantic-release`:

```yaml
- run: npm install -D git+https://github.com/PanoMC/semantic-release-pano.git
```

Это единственная правка рабочего процесса, которая нужна аддону, публикующемуся на Маркетплейсе; рабочий процесс `pano-plugin-faq`, из которого скопирован этот конфиг, имеет ровно этот шаг.
:::

Поле за полем:

| Поле | Что оно делает |
|---|---|
| `@semantic-release/github` `assets` | Прикрепляет ваш `build/libs/*.jar` (и `LICENSE`) к релизу GitHub. Этот шаг выполняется **до** плагина Pano, чтобы актив существовал, когда он понадобится `useGitHubLink`. |
| `file` | Путь к собранному jar. `${version}` подставляется версией релиза. |
| `panoVersion` | Версия платформы Pano, на которую нацелен этот релиз. |
| `useGitHubLink` | `true` = не перезагружать jar; вместо этого указать Маркетплейсу на jar, уже прикреплённый к релизу GitHub (плюс его хеш SHA-256). Идеально для бесплатных аддонов — без дублирующей загрузки. Премиум-аддоны загружают jar напрямую. |
| `configs[]` | По одной записи на канал. Каждая говорит, **на какой Маркетплейс публиковать** и **с каким токеном**, ограничиваясь `branches`. |
| `resourceId` | Ваш ресурс на Маркетплейсе — для аддонов это ваш `pluginId` (см. ниже). |
| `panoUrl` | API Маркетплейса: `https://api-dev.panomc.com` для канала `dev`, `https://api.panomc.com` для `main`. |
| `tokenVar` | Имя секрета GitHub, хранящего API-токен: `PANO_TOKEN` для dev, `PANO_PROD_TOKEN` для production. |
| `branches` | Ограничивает конфиг одним каналом, так что push в `dev` никогда не трогает production-Маркетплейс (а отсутствующий `PANO_PROD_TOKEN` не завалит сборку `dev`). |

::: tip Зачем два `configs`
Разделение по веткам означает, что вы можете отрепетировать релиз на `api-dev.panomc.com` из вашей ветки `dev`, а затем отправить ровно тот же код на `api.panomc.com` из `main` — каждый со своим состоянием ресурса и токеном. Если вы публикуетесь только на один Маркетплейс, оставьте только конфиг `main`.
:::

## Публикация на официальном Маркетплейсе Pano

Маркетплейс на [panomc.com](https://panomc.com) — это место, где владельцы серверов находят и устанавливают аддоны прямо из своей панели. Публикация там занимает три шага: создать ресурс, создать API-токен, а затем позволить автоматизации загружать версии.

### 1. Создайте ресурс

1. Зарегистрируйтесь (или войдите) на **panomc.com**.
2. Откройте **Create Resource** и выберите тип **Plugin**.
3. Выберите категорию, заполните заголовок и описание.
4. Выберите ценообразование: **бесплатно** или **платно**, если планируете продавать.

::: tip ID ресурса — это ваш ID плагина
Темы используют случайный UUID для своего ID ресурса на Маркетплейсе, отдельный от `id` их пакета. **Аддоны — нет.** `resourceId` вашего аддона на Маркетплейсе установлен равным **ровно вашему `pluginId`** — для Shoutbox это `pano-plugin-shoutbox`. Именно поэтому в `configs[]` выше используется `"resourceId": "pano-plugin-shoutbox"`, а не UUID. `pluginId` — это единая идентичность, используемая повсюду: имя каталога данных, префикс узлов прав доступа, сегмент URL UI и ресурс на Маркетплейсе. См. [Конфигурацию манифеста](/ru/addon/manifest/).
:::

### 2. Создайте API-токен

1. На panomc.com откройте **Profile → Settings → API Tokens** и нажмите **Create**.
2. Скопируйте токен немедленно — он показывается **только один раз**, в модальном окне сразу после создания.
3. В вашем GitHub-репозитории перейдите в **Settings → Secrets and variables → Actions** и добавьте токен как секрет репозитория, названный так, чтобы совпадать с вашим `tokenVar`:
   - `PANO_TOKEN` для канала `dev`,
   - `PANO_PROD_TOKEN` для канала `main`.

::: warning Никогда не коммитьте токен
API-токен даёт права публикации на ваш ресурс. Храните его только как секрет GitHub (или локальную переменную окружения). Никогда не помещайте его в `.releaserc.json`, коммит или любой файл в репозитории.
:::

Сам релиз GitHub использует отдельный токен, `TOKEN_GITHUB`, который рабочий процесс boilerplate читает (как `secrets.TOKEN_GITHUB`) в нескольких местах, включая начальный сухой прогон версии. Встроенный `GITHUB_TOKEN` от GitHub **не** доступен под этим именем, поэтому вы должны создать персональный токен доступа (Personal Access Token) и сохранить его как секрет репозитория с именем `TOKEN_GITHUB` — иначе каждый прогон релиза упадёт на первом шаге semantic-release. (Как вариант, отредактируйте рабочий процесс, чтобы он читал `secrets.GITHUB_TOKEN`.)

### 3. Запушьте и наблюдайте за публикацией

С созданным ресурсом, добавленным как секрет токеном и `.releaserc.json` на месте публикация — это просто:

```bash
git push origin dev    # или main
```

Рабочий процесс выполняется, semantic-release вычисляет версию, создаётся релиз GitHub с прикреплённым jar, а `@PanoMC/semantic-release-pano` регистрирует эту версию на Маркетплейсе. Владельцы серверов увидят обновление в своей панели под **Панель → Аддоны**.

## Ручное распространение

Вам не обязательно использовать Маркетплейс. Поскольку релизный jar полностью самодостаточен, любой может установить его напрямую:

- Прикрепите jar к релизу GitHub и поделитесь ссылкой, или
- Передайте `.jar` владельцу сервера, чтобы он загрузил его через **Панель → Аддоны → Загрузить**.

Разница, которую видит пользователь: аддоны, скачанные с Маркетплейса, помечены как **проверенные (verified)**, тогда как загруженные вручную jar — нет. Для публичного аддона Маркетплейс настоятельно предпочтителен — он бесплатно даёт вам доставку обновлений, страницу в магазине и значок проверки.

## Премиум-листинги

Продажа вашего аддона работает через тот же процесс релиза, плюс ключ лицензии времени сборки и проверка лицензии времени выполнения, вшитая в ваш код. Сборка принимает дополнительные флаги лицензии, описанные в [Конфигурации манифеста](/ru/addon/manifest/) (`-PlicenseServer`, `-PpanoLicensePublicKey` и переменную окружения `PANO_LICENSE_PUBLIC_KEY`); полное руководство по премиуму пока не является частью этих туториалов.
