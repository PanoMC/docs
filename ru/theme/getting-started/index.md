# Начало работы

Этот раздел охватывает всё, что нужно для создания темы Pano. Тема управляет **внешним видом и восприятием** публичного сайта — макетами, цветами, типографикой и разметкой, — тогда как бэкенд, аутентификация, среда выполнения плагинов и загрузка данных берутся на себя движком.

## Что такое тема сегодня

Современная тема Pano — это **тонкий пакет**, построенный поверх движка **`@panomc/theme-core`**, опубликованного в npm (текущая линейка версий `1.0.0-dev.x`). Движок поставляет потоки аутентификации, среду выполнения плагинов, загрузку данных и конвейер сборки в виде зависимости — вы используете его и обновляете командой `bun update`, как любой другой пакет.

Репозиторий вашей темы обычно занимает всего несколько сотен строк: токены дизайна, необязательные переопределения представлений (view) и метаданные. Всё сложное уже живёт внутри движка.

> **`vanilla-theme` — это встроенная СИСТЕМНАЯ тема, а не шаблон.** Не **форкайте и не копируйте** её. Она защищена и управляется Pano внутренне. Вместо этого создавайте новую тему с помощью генератора шаблона (scaffolder), описанного ниже.

## Требования

Перед началом убедитесь, что у вас есть:

- Установленный **Bun** (фронтенды Pano используют Bun, а не npm/pnpm).
- **Локально запущенный экземпляр Pano** — сервер разработки вашей темы проксирует API-запросы к нему.
- Базовое знакомство с **Svelte 5** и **SCSS** (нужно только когда вы выходите за пределы токенов).

## Создание темы

Создайте новую тему с помощью CLI `theme-core`:

```sh
bunx theme-core new my-theme
```

Эта команда генерирует **шаблон из 17 файлов** — манифест, конфигурацию, стилевые токены, shim-файлы хуков и каркас SvelteKit. Затем установите зависимости и запустите:

```sh
cd my-theme
bun install
bun run sync          # генерирует мосты маршрутов/lib из движка
bun run dev           # сервер разработки против запущенного бэкенда Pano
```

> **Если `bun install` зависает на шаге «Resolving…»**, прервите его и повторите с флагом:
> ```sh
> bun install --backend=copyfile
> ```

Сервер разработки проксирует API-запросы к вашему локальному бэкенду Pano. Укажите адрес в `.env`:

```sh
# .env
VITE_API_URL=http://localhost:8088/api
```

## Что принадлежит вам, а что генерируется

Понимание того, какие файлы ваши, а какие пересоздаются, — ключ к безболезненным обновлениям.

**Ваши — редактируйте свободно, храните в репозитории:**

| Путь | Назначение |
|---|---|
| `manifest.json` | id, заголовок, версия, автор, `panoVersion`, скриншоты |
| `theme.config.js` | переопределения представлений + расширения схемы настроек |
| `src/styles/tokens.scss` | ваши токены дизайна (цвета, радиусы, шрифты, тени) |
| `src/styles/style.scss` | сначала токены, затем SCSS движка, затем ваш CSS |
| `src/views/` | ваши переопределения представлений (только те, что вы извлекли) |
| `lang-overrides/` | только изменяемые вами ключи i18n (глубокое слияние, только добавление) |
| `static/` | ваши ресурсы (assets) |

**Генерируемые — никогда не редактируйте вручную (`bun run sync` пересоздаёт их):**

| Путь | Назначение |
|---|---|
| `src/routes/` | shim-файлы маршрутов, сгенерированные из движка |
| `src/lib/` | сгенерированные мосты + заглушки host-provides SDK |
| `lang/` | базовые языковые файлы (переопределяйте через `lang-overrides/`) |

> Редактирование сгенерированных файлов вручную означает, что ваши изменения исчезнут при следующем `sync` или обновлении движка. Помещайте каждое изменение в файлы, которые принадлежат вам.

## Уровень 1 — Токены (перекраска без обслуживания)

Простейшая тема не меняет ничего, кроме токенов. Файл `src/styles/tokens.scss` поставляется как **закомментированное меню каждой переменной движка**. Раскомментируйте нужные и измените их значения:

```scss
// src/styles/tokens.scss
$color-primary:    #ff5722;
$color-background: #0f1115;
$radius-base:      12px;
$font-family-base: "Inter", sans-serif;
```

Каждая переменная движка объявлена как `!default`, поэтому ваше значение всегда побеждает. Тема, состоящая только из токенов, **не требует обслуживания вечно** — она продолжает работать через minor- и даже major-версии движка без правок. Уже одно это даёт заметно отличающуюся тему.

## Уровень 2 — Представления (собственная разметка)

Когда токенов недостаточно и нужна другая разметка, переопределите **представление (view)**. Выведите список всех переопределяемых представлений и их пропсов:

```sh
bunx theme-core list-views
```

Всего есть **26 переопределяемых представлений**. Чтобы настроить одно, извлеките его (eject):

```sh
bunx theme-core eject-view LoginView
```

Эта команда копирует представление движка по умолчанию в `src/views/LoginView.svelte` и регистрирует его в `theme.config.js`. Заголовок файла (header) документирует **каждый пропс** — хранилища (store) приходят как объекты хранилища (используйте `$store`), а действия (action) — как функции.

Обрамление страницы (`Navbar`, `Header`, `Footer`) и компоненты, обращённые к плагинам (`LoginFormBody`, `RegisterForm`, `Pagination`, …), переопределяются по отдельности точно так же — вам не нужно извлекать весь макет, чтобы перекрасить навбар.

> **Сохраняйте каждую точку монтирования плагинов.** Переопределённое представление **обязано** сохранить слоты `<ViewComponent>` и маркеры `<Hook>`, которые монтировало представление по умолчанию. Если вы удалите хоть один, установленные плагины молча исчезнут — и `bun run check` завершится ошибкой ещё до того, как вы соберёте тему.

## Дополнительные настройки темы

Страница настроек темы управляется сопоставлением вкладка → ключ настройки. Если ваше переопределённое представление отображает дополнительные поля ввода — новые ключи или даже целую новую вкладку — объявите их в `theme.config.js`, чтобы панель их **сохраняла и сбрасывала** (иначе они отображаются, но никогда не сохраняются):

```js
// theme.config.js
export default {
  views: {
    ThemeSettingsView: () => import("./src/views/ThemeSettingsView.svelte"),
  },
  settingsSchema: {
    // Только добавление: ключи ДОБАВЛЯЮТСЯ во вкладку (новая вкладка создаётся,
    // если её нет). Базовый ключ нельзя удалить или переместить.
    tabs: {
      header: ["heroSubtitle", "heroSubtitleVisibility"],
      "support-page": ["supportPageDiscordLink"],
    },
    // Необязательно: вкладка, на которой открывается страница. Требуется, только
    // если ваше представление не отображает базовую вкладку по умолчанию ("general").
    defaultTab: "logo",
  },
};
```

Правила: ключ может находиться ровно в **одной** вкладке (сохранение/сброс выполняются по вкладкам), а `defaultTab` должен быть реальной вкладкой. Ключ, который вы только *читаете* в разметке (без поля ввода в представлении настроек), не требует записи в схеме.

## Публикация темы

Сборка и упаковка воспроизводимы по своей природе (reproducible):

```sh
bun run build        # воспроизводимая сборка (версия kit закреплена, детерминирована)
bun run check        # проверяет контракт — завершается ошибкой при любом нарушении
bun run package      # детерминированный zip
```

Затем установите полученный `.zip` через **Panel → Просмотр → Темы → Установить тему**.

**Премиум-темы:** **sha256 zip-архива является идентификатором лицензии**. Лицензионный барьер **пропускается под `vite dev`**, чтобы вы могли свободно разрабатывать, и **применяется в production-сборках**.

`bun run check` применяет контракт перед публикацией:

- `svelte` закреплён точно на версии core (расхождение молча выбрасывает плагины)
- каждое зарегистрированное представление существует и является известным именем контракта
- переопределённые представления сохраняют каждый slot/hook, монтируемый по умолчанию
- `lang-overrides/*.json` разбираются и сливаются только добавлением
- `settingsSchema` корректен по форме, только добавляет, а его `defaultTab` реален
- `manifest.json` содержит обязательные ключи, а его `id` не равен `vanilla-theme`

## Обновление движка

Обновление — это цикл из трёх команд:

```sh
bun update @panomc/theme-core && bunx theme-core sync && bun run build
```

- **Темам только из токенов** больше ничего не нужно — это вся миграция, **даже через major-версии**.
- **Переопределения представлений** не затрагиваются minor-версиями (пропсы только добавляются, никогда не изменяются и не удаляются). При **major**-версии внимания могут потребовать только те представления, что вы переопределили — `bun run check` перечислит каждое нарушение контракта, а список изменений движка задокументирует изменения пропсов по каждому представлению.

Вот и вся модель: оставайтесь на токенах ради темы без обслуживания, спускайтесь к представлениям только там, где нужна собственная разметка, и позвольте движку нести остальное.
